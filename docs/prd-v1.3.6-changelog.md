## 一、本次反馈摘要（2026-06-08 客户第四次反馈）

承蒙木辛日匀总在 V1.3.5 之后第四次抽时间再为本项目调整"厂商对账"环节，本次反馈我方**严格按"灵魂八字 + 善良设计 + 一数到底"**收口落地——**同样不另收费、不延期**。

> 客户原话：*"因为是**本地部署**，所以厂商没有微信操作这一部。通知厂商可选择**邮件推送/短信推送**，需要**采购线下拿对账单跟厂商确认**，然后将厂商**签字确认后拍照上传系统**流转至财务，其余流程不变。"*

### 1.1 三大核心动作

| 维度 | 客户口径 | 我方落地 |
|------|----------|---------|
| **取消项 1** | **删除厂商微信对账**（V1.3.5 描述的"厂商通过微信/短信链接确认"） | 通知渠道改"邮件推送 / 短信推送"；删除任何"微信链接 / 厂商扫码确认"描述 |
| **取消项 2** | **删除厂商线上对账模块** | 厂商无系统账号 → 无任何"线上对账"入口；改为"线下 + 拍照上传" |
| **新增项 1** | **邮件推送 / 短信推送**通知厂商 | Nacos `app.outsub.notify-channel: ['email', 'sms']` 多渠道可配；SMTP + 阿里云/腾讯云短信 SDK |
| **新增项 2** | **采购线下拿对账单 + 厂商签字 + 拍照上传** | E6-S1 整章改版为"月度对账 + 线下 + 拍照上传"；签字扫描件作为对账法律凭证沉淀进系统 |

### 1.2 业务现实确认（V1.3.6 客户原话落地）

> "因为是**本地部署**，所以厂商没有微信操作这一部"

⇒ 我方在 PRD 中**明确"本地部署场景下厂外 0 IT 负担"**——不开放任何供应商系统账号，所有通知走邮件/短信，所有对账凭证通过"线下签字 + 拍照上传"沉淀进系统。

> "通知厂商可选择**邮件推送/短信推送**"

⇒ 我方设计**Nacos 多渠道可配**（email / sms / 站内信），默认 `['email', 'sms']`；客户可按厂商偏好定制（部分老厂商只接受短信）。

> "需要**采购线下拿对账单跟厂商确认**，然后将厂商**签字确认后拍照上传系统**流转至财务"

⇒ 我方设计**"四阶段对账闭环"**：
- 阶段 1（系统）：采购 PC 端生成对账单 PDF → 邮件发厂商
- 阶段 2（线下）：采购带纸质对账单去厂商处 → 厂商核对 → 厂商签字盖章
- 阶段 3（系统）：采购拍照/扫描上传系统 → 录入对账确认日期 → 状态置"对账已确认"
- 阶段 4（系统 + 财务）：系统自动通知财务 → 财务校验"双方金额一致" → 生成付款申请 → 走原有双签流程

## 七、灵魂一致性自评（PM 范蠡）

> **范蠡本次 V1.3.6 改版是否保持了"一码到底、一数到底"灵魂？**
>
> **一句话回答**：**"一数到底"从"线上对账"收敛为"线下签字 + 系统留痕"——灵魂没变，反而更扎实；"一码到底"维持 V1.3.5 收敛（3 类 + 委外单码），未再扩张**。
>
> **"一数到底"如何在"线下对账"下仍能贯穿？**
>
> 答：**"一数到底"从来不靠"全线上"维持，而靠"每一份数据可被某个岗位负责"维持**。V1.3.6 把"对账"拆为 4 个阶段：
> - 阶段 1（系统动作，采购员负责）：生成对账单 PDF + 邮件推送
> - 阶段 2（线下物理动作，厂商 + 采购员负责）：核对 + 签字盖章
> - 阶段 3（系统动作，采购员负责）：拍照/扫描上传
> - 阶段 4（系统动作 + 财务动作，财务员负责）：校验金额 + 生成付款申请
>
> 每一个阶段都有明确的"责任人 + 系统动作 + 物理动作"——"线下"不等于"无据可查"，"系统留痕" = **签字扫描件 AES-256 入库 + 5 年保留 + 全审计**。
>
> **为什么不让厂商装 APP / 扫微信？**
>
> 答：**善良设计**——本地部署场景下，厂外是"客人"，客人不应该为"进我家门"装 APP。V1.3.6 让"邮件 + 短信 + 纸质签字"成为合规闭环，对厂外**零 IT 负担**。这不是"妥协"，而是"对厂外组织的尊重"。
>
> **自评分数**：9.3 / 10（V1.3.5 为 9.2，+0.1）
>
> **加分项**：
> - **线下对账 + 拍照上传**：法律凭证沉淀进系统，且对厂外 0 IT 负担
> - **付款触发条件升级**：避免"系统说对账了，财务没看到附件就付款"的合规漏洞
> - **签字件 AES-256 + 5 年保留**：财务合规 + 诉讼时效内可追溯
>
> **扣分项**：
> - 厂商资料迁移（旧厂商邮箱缺失）需运营介入补填——已列入 §8.4 #13 开放问题
> - 短信网关需提前签约（阿里云/腾讯云）——已列入 §8.4 #14 开放问题

## 十二、现场操作示例（V1.3.6 更新版）— 一单全流程

> **场景**：2026 年 6 月，A 表处厂承接我司"第 3 道表处"工序委外，6/15 完工回厂入库，本月累计 10 单。

### 步骤 1：委外下单（E5-S4 + 邮件/短信通知）

**操作员**：生管员 小张
**时间**：2026-06-08 09:00
**操作**：
1. 打开 ERP → 工单 GD-20260608-001 详情 → 选第 3 道"表处" → 委外
2. 选厂商"A 表处厂" → 系统自动带出历史价 8 元/件（中位数）
3. 录入数量 100 件 → 总金额 800 元
4. 提交委外订单 → 审批通过 → 生成委外单 **WW-20260608-001**

**系统自动动作**：
- 邮件推送至 A 表处厂"业务邮箱"（`contact@abiaochu.com`）：
  > 主题：【××精密机械】新委外订单 WW-20260608-001
  > 正文：（详见 §五.1 邮件模板）
  > 附件：委外订单详情 PDF（含二维码 WW-）
- 短信推送至 A 表处厂"业务联系人电话"（`138-0000-0000`）：
  > 【××机械】新委外单 WW-20260608-001: 第3道表处, 100件, 交期6/15, 详情见邮件。联系采购王小王 138-xxxx。

### 步骤 2：厂商加工 + 送货回厂

**操作员**：A 表处厂 加工 + 送货
**时间**：2026-06-08 ~ 2026-06-15
**操作**：A 表处厂加工完毕 → 派车送货 → 随车带**纸质送货单**（贴有 WW- 委外单二维码）

### 步骤 3：仓管到货扫码（V1.3.5 E12-S2 · 维持）

**操作员**：仓管员 王仓管
**时间**：2026-06-15 14:30
**操作**：
1. 仓管 APP 打开"仓储 → 到货扫码"
2. 扫送货单上的 WW- 委外单二维码

4. 实收数量 100 / 重量 50kg / 拍 2 张外观照片
5. 点"确认" → 状态 `SHIPPING → PENDING_INSPECTION`
6. 自动通知生管（APP + 企业微信）+ 品质（APP + 企业微信）
