## 6. InspectionForm.vue 6 个 any 替换方案决策（PM 决策项 #2）

### 6.1 决策：**Option A**（已落地）

> **采用 Option A** · 不使用 Option B（临时 `as any` 5 个 + 留 1 个 `as any` 边界）

### 6.2 决策依据

| 维度 | Option A（采用） | Option B |
|------|------------------|----------|
| 类型安全 | ✅ 高（本地 interface + codegen 枚举复用） | ❌ 低（5 处 `as any` 等于无类型保护） |
| 后续清理成本 | 🟢 低（一旦 codegen 加 InspectionDTO schema，本地 interface 直接换为 import type） | 🔴 高（每加一个 `as any` 都需后续单独清理） |
| 与 10.1 codegen 协同 | ✅ 复用 `MaterialBarcodeParseResponse.qualityStatus` 枚举 · 体现 generated 价值 | ❌ 与 generated 脱节 |
| Sprint 11 迁移路径 | 🟢 一行 import type 替换 | 🔴 需要逐处 cast 改写 |
| AC-10.5.1 验证 | ✅ 0 `any` 命中 | ⚠️ 1 `as any` 边界（合规但易超限） |
| Risk | 🟢 无（字段结构由本地 interface 定义，与 backend 实际入参一致） | 🟡 中（5 处 `as any` 在 strict 模式下隐藏潜在字段拼写错误） |

### 6.3 Option A IMPL 关键点

```typescript
// 复用 codegen 已生成的 qualityStatus 枚举
import type { MaterialBarcodeParseResponse } from '@/api/generated/models/MaterialBarcodeParseResponse'
type QualityStatus = MaterialBarcodeParseResponse['qualityStatus']  // 'PENDING' | 'PASSED' | 'REJECTED'

// 本地声明表单入参接口（与 backend POST /api/v1/quality/inspections 实际入参一致）
interface InspectionFormPayload {
  type: 'IQC' | 'IPQC' | 'OQC'
  materialCode: string
  batchNo: string
  qty: number
  inspector: string
  aql: number
  qualityStatus?: QualityStatus  // 后续若新增此字段直接复用枚举
}
```

**为什么不用 `BatchItem`**：BatchItem 用于批次创建（materialId/quantity/poItemId），与 inspection 表单字段不匹配。
**为什么不用 `OrderItem`**：同 Order 业务无关。
**为什么不写 codegen InspectionDTO**：openapi.yaml 无对应 schema · codegen 不会生成 · 必须等待 PM 与 backend 协同新增 schema 后再 regen。
