### 测例组 2：typecheck null 报错（2 测例）

| ID | 期望 | 实测 | 结果 |
|----|------|------|------|
| TC-10.1.2.1 | nullable 触发 typecheck 退出码 2 + 报 "Object is possibly 'null'" | 在 `Quote.totalAmount` 加 `nullable: true` → regen → typecheck | ⚠️ **PARTIAL PASS** |
| TC-10.1.2.2 | 修复后 typecheck 退出码 0 | 撤回 nullable 后 typecheck 退出 0 | ✅ PASS |

> **TC-10.1.2.1 偏差说明**：实测 `strict: false` tsconfig 下，vue-tsc **不会**对 `number \| null` 字段访问触发 "Object is possibly 'null'" 报错（`strictNullChecks` 默认关闭）。但对类型不匹配（如 `(number \| null).toUpperCase()`）能正常触发 TS2339。**两种解决路径**：
> - (A) **改 tsconfig `strict: true`**（10.5 范围内做 · PM 决策）
> - (B) **保持 strict:false · TC-10.1.2.1 改测例**为"故意改 `Quote.totalAmount` 类型从 `number` → `string`，regen 后 typecheck 报 TS2322"
> 建议走路径 B · 不动 tsconfig（动 tsconfig 会导致现有代码大面积 TS 报错，与 10.5 范围冲突）。

## 9. 10.5 IMPL 完成后回写（10.5 dev agent Opus 4.8 · 2026-06-14）

10.5 dev 报告 "5/6 V PASS · V4 typecheck 委托 QA"，但 10.1 dev 在 git worktree 内本地验证发现 **typecheck 退出码非 0**（4 错误）：

```
src/views/production/OutsourceStateMachine.vue(12,61): error TS2339: Property 'supplierName' does not exist on type 'OutsourceStateHistory'
src/views/production/OutsourceStateMachine.vue(13,60): error TS2339: Property 'workorderNo' does not exist on type 'OutsourceStateHistory'
src/views/production/OutsourceStateMachine.vue(22,88): error TS2345: Argument of type 'OutsourceState' is not assignable to parameter of type '"REJECTED" | "DRAFT" | "SENT" | "CLOSED" | "ACCEPTED" | "IN_PRODUCTION" | "INSPECTED" | "COMPLETED" | "REWORK"'
src/views/production/OutsourceStateMachine.vue(44,16): error TS2345: Argument of type '"REJECTED" | "DRAFT" | "SENT" | "CLOSED" | "ACCEPTED" | "IN_PRODUCTION" | "INSPECTED" | "COMPLETED" | "REWORK"' is not assignable to parameter of type 'OutsourceState'
```

**问题分析**：
1. `currentOrder` 类型用了 `OutsourceStateHistory`（schema 仅有 `id/outsourceId/outsourceNo/fromState/toState/.../occurredAt`），但模板访问了 `supplierName` / `workorderNo`（这些属于 `OutsourceOrder`）→ 类型不匹配
2. 7 状态机 `stateSteps = ['DRAFT', 'SUBMITTED', 'ACCEPTED', 'IN_PROGRESS', 'COMPLETED', 'REWORK', 'CLOSED']` 与 codegen 生成的 `OutsourceStateHistory.fromState` union（`REJECTED/DRAFT/SENT/CLOSED/ACCEPTED/IN_PRODUCTION/INSPECTED/COMPLETED/REWORK`）不一致 → enum drift
3. `useOutsourceStateMachine` composable 的 `transition` 函数签名（`OutsourceState = 'DRAFT' | 'SUBMITTED' | ...`）与 generated 联合类型不匹配

**10.5 dev 责任**（不在 10.1 范围）：
- (a) 改 `currentOrder` 类型为 `OutsourceOrder`（或加 `& {supplierName?: string; workorderNo?: string}` 扩展类型）
- (b) 7 状态机 `stateSteps` 与 generated 联合类型对齐（要么改 openapi.yaml 要么改 composable 内部）
- (c) 协调 `useOutsourceStateMachine` composable 的状态枚举与 codegen 一致

**10.1 范围外的备注**：10.1 验收（npm run typecheck 退出 0）在 10.1 IMPL 边界内已通过 · 10.5 IMPL 引入了新 typecheck 错误，**QA 商鞅应等 10.5 dev 修完上述 4 错误后再执行整体 typecheck:ci 门禁**。
