## 一、本次反馈摘要（2026-06-08 客户第五次反馈）

承蒙木辛日匀总在 V1.3.6 之后第五次抽时间再为本项目做"职责边界 + 流程描述边界 + 通知渠道"三件套的精细调整，本次反馈我方**严格按"灵魂八字 + 善良设计 + 一数到底"**收口落地——**同样不另收费、不延期**。

> **客户三大原话**：
> 1. *"由**生管**决定哪些工序自制、哪些工序委外，然后委外到**哪家由采购决定**。"*
> 2. *"对账流程采用：采购 PC 端'委外对账'模块选对账月份 + 厂商 → 系统自动汇总该月所有 WW- 单 → 生成对账单 PDF → 点'发送对账单邮件'，**线下流程不去涉及**。"*
> 3. *"通知厂商改为 **163 邮箱邮件通知**。"*

### 1.1 三大核心动作

| 维度 | 客户口径 | 我方落地 |
|------|----------|---------|
| **职责分离** | **生管决定自/委，采购选厂商** | V1.3.6 之前"业务/生管划分、采购选厂商"模糊描述明确为两条线不交叉；生管只勾工序归属不选厂商，采购只选厂商不决定自/委 |
| **PRD 边界** | **对账流程不含"线下"描述** | 删除"采购带纸去厂商处 → 厂商核对 → 厂商签字"段落；**厂商线下流程不在 PRD 描述**；保留"采购上传签字扫描件"作为合规凭证 |
| **通知渠道** | **单一 163 邮箱** | Nacos `app.outsub.notify-channel = ['email_163']`；删除短信渠道；删除短信模板管理；厂商资料电话改选填 |

### 1.2 业务现实确认（V1.3.7 三大客户原话落地）

> "由**生管**决定哪些工序自制、哪些工序委外，然后委外到**哪家由采购决定**"

⇒ 我方在 PRD 中**新增 §3.4 委外加工管理 · 工序分配职责**章节，明确"工序划分 = 生产计划问题（生管）/ 厂商选择 = 供应链问题（采购）"——不越权、不甩锅；E5-S4 改版为"生管勾选工序归属 → 系统生成待委外清单 → 采购为每道工序选厂商"。

> "对账流程采用……线下流程不去涉及"

⇒ 我方在 PRD 中**删除"采购线下打印对账单 → 采购带纸去厂商处 → 厂商核对 → 厂商签字盖章 → 采购拍照上传"整段**——只描述"采购 PC 端点'发送对账单邮件' → 系统发邮件 → 厂商签完字后通过任意方式回传给采购 → 采购上传扫描件"；**厂商线下流程（核对/签字/回传方式）不在 PRD 描述**。

> "通知厂商改为 **163 邮箱邮件通知**"

⇒ 我方设计**单 163 邮箱 SMTP 渠道**（`smtp.163.com:465 SSL`），授权码从环境变量 `EMAIL_163_AUTH_CODE` 注入；删除阿里云/腾讯云 SMS SDK 依赖；删除短信模板管理模块；厂商资料电话从 V1.3.6 必填改 V1.3.7 选填（短信不再需要）。

## 八、灵魂一致性自评（PM 范蠡）

> **范蠡本次 V1.3.7 改版是否保持了"一码到底、一数到底"灵魂？**
>
> **一句话回答**：**"一数到底"从"一个角色包打天下"收敛为"每个角色只对自己的数据负责"——灵魂没变，反而更清；"一码到底"维持 V1.3.5 收敛（3 类 + 委外单码），未再扩张**。
>
> **"一数到底"如何在"职责分离"下仍能贯穿？**
>
> 答：**"一数到底"从来不靠"全流程一个人做完"维持，而靠"每一份数据可被某个岗位负责"维持**。V1.3.7 把"工序分配"拆为 2 个独立动作：
> - 动作 1（生管负责）：勾选工单每道工序归属（厂内/委外），**不指定厂商**
> - 动作 2（采购负责）：接收生管的待委外清单，**为每道工序选厂商**
>
> 两个动作有明确的"责任人 + 系统动作 + 数据落点"——"职责分离"不等于"数据断裂"，"系统留痕" = **工单工序归属记录 + 委外单 WW- + 163 邮箱发送日志 + 签字扫描件 AES-256**。
>
> **"PRD 边界"为什么重要？**
>
> 答：**善良设计**——PRD 应该描述"我方系统做什么"，不应该描述"厂商线下怎么做"。V1.3.7 把"采购带纸去厂商处 → 厂商核对 → 厂商签字"这一段**划出 PRD 边界**——厂商可以用微信 / 邮件 / 派车取回 / 任何方式把签字版给采购，PRD 不强制；这样既尊重了厂商的灵活操作习惯，也避免 PRD 因为描述线下流程而失去对"系统能力"的聚焦。
>
> **"单一 163 邮箱"为什么是收敛而非妥协？**
>
> 答：**极致简化**——V1.3.6 的"邮件 + 短信"看似"多渠道冗余"，实则引入了短信网关签约 / 充值 / 模板审核 / 签名等额外复杂度；V1.3.7 收敛为单一 163 邮箱，**减少了 1 个第三方依赖、1 套 SDK 集成、1 个网关签约**——把"少一个故障域"做到极致。
>
> **自评分数**：9.4 / 10（V1.3.6 为 9.3，+0.1）
>
> **加分项**：
> - **工序分配职责分离**：每个角色只做自己擅长的事，灵魂更清
> - **PRD 边界明确**：只描述"我方系统做什么"，不越界描述"厂商线下怎么做"
> - **单一 163 邮箱**：减少 1 个第三方依赖、1 套 SDK 集成、1 个网关签约
> - **邮件投递保障三重防线**：重试 + 额度监控 + 发送日志，单一渠道也能可靠
>
> **扣分项**：
> - 163 邮箱单日额度 500 封（普通账号）需监控 / 升级 VIP——已列入 §10 #18 开放问题

## 十一、现场操作示例（V1.3.7 更新版）— 一单全流程

> **场景**：2026 年 6 月，A 表处厂承接我司"第 3 道表处"工序委外，6/15 完工回厂入库。

### 步骤 1：生管完成工序划分（V1.3.7 新增阶段）

**操作员**：生管员 小张
**时间**：2026-06-08 09:00
**操作**：
1. 打开 ERP → 工单 GD-20260608-001 详情 → "工序划分" Tab
2. 5 道工序逐个勾选：工序 1 车 [厂内] / 工序 2 CNC [厂内] / 工序 3 表处 [委外] / 工序 4 钳工 [厂内] / 工序 5 包装 [厂内]

### 步骤 2：采购为工序 3 选厂商 + 创建委外单（V1.3.7 改版）

**时间**：2026-06-08 09:15
