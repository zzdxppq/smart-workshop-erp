## 4. 灰度发布 4 周计划

### 阶段 1：影子表 + 双写（4 周 · IMPL 后立即启动）

- ✅ V49 已建 crm_batch + crm_batch_shadow
- ✅ BatchService 双写（每次到货同时写两表）
- ✅ xxl-job `batchShadowCompareHourly` 每小时对比
- ✅ 不一致率 > 0.1% 触发告警
- 监控：`SELECT COUNT(*) FROM crm_batch; SELECT COUNT(*) FROM crm_batch_shadow;`

### 阶段 2：切流 50%（1 周 · FAT 通过后）

- Nacos 配置 `app.batch.read-source=NEW` 设 50% 流量走新表
- 老路径保留作为 fallback
- 监控 P95 性能 + 不一致率

### 阶段 3：全量切流（1 周）

- Nacos 配置 `app.batch.read-source=NEW` 设 100%
- 监控 P95 + 不一致率
- 如不一致率 > 0.1% 立即回滚

### 阶段 4：清理（1 周）

- DROP TABLE crm_incoming（V1.3.7 老表）
- 删除 BatchService 老路径代码
- DROP TABLE crm_batch_shadow（保留 30 天后删除）

## 5. 回滚步骤

如需回滚 V1.3.8 → V1.3.7：

```bash
# Linux/macOS
./docs/dev/scripts/deploy-v1.3.8.sh --rollback

# Windows
.\docs\dev\scripts\deploy-v1.3.8.ps1 -Rollback
```

**自动回滚会做**：
1. `docker compose down` 停服
2. 恢复 MySQL 数据库（/tmp/erp_v137_backup.sql）
3. git checkout v1.3.7 -- backend/db/migrations/V49-V52.sql
4. 重新构建镜像 + 启动

**手动回滚（更彻底）**：
```bash
docker compose down
git checkout v1.3.7
cd backend && mvn clean package -DskipTests
cd deploy && docker compose up -d --build
```
