## 三、仓储模块（含仓管 Web 端规划）

### 3.1 仓管 Web 端功能规划（当前缺失）

| 序号 | 问题描述 | 当前系统现状 | PRD 要求 | 整改动作 | 优先级 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **3.1** | **Web 端缺少仓管核心功能** | 目前只有“物料→库存”和“库存预警”，无仓管专用工作台 | 仓管需要PC端完成：批量入库、打印标签、库存查询、批次追溯、盘点 | **新增一级菜单“仓储”**，包含以下子菜单：<br>①仓储总览（待入库数量/待上架/低库存预警卡片）<br>②入库单（待处理列表 → 批量执行入库 → 打印标签）<br>③出库单<br>④库存查询（按物料/库位/批次）<br>⑤批次列表（现有）<br>⑥库存预警（现有）<br>⑦盘点单 | **P0** |
| **3.2** | **PC 端“批量入库+打印标签”功能缺失** | 只有APP简易入库，无PC批量处理入口 | E4-S1 / V1.3.8：仓管应能在PC端批量勾选待入库单，执行入库并批量打印标签 | ①“入库单（待处理）”列表：展示所有状态=待入库的单据；<br>②支持多选 → 点击“批量执行入库”；<br>③弹窗确认良品数 → 系统自动生成批次码；<br>④点击“打印本次标签（X张）”批量打印；<br>⑤APP端保留单件快速入库入口，但不支持打印标签（提示“请至PC端补打”） | **P0** |
| **3.3** | **条码提前生成（图纸列表）** | 图纸列表有“生成条码”按钮，生成 `BC20260619-0002` 流水号 | V1.3.8 / E4-1-1：条码只能在**仓库入库时生成**，格式为复合码 `{物料编码}-BATCH-{日期}-{流水}` | ①**删除图纸列表所有“生成条码”按钮**；<br>②条码生成功能移至“仓储→入库单”页面，仅在“执行入库”时出现 | P0 |
| **3.4** | **条码编码规则错误** | 条码号显示 `WL-DEMO-STEEL-001` 或 `BC20260619-0001`，不含BATCH | V1.3.8 Story 3.2：条码格式必须为 `{物料编码}-BATCH-{YYYYMMDD}-{seq:4}` | ①修订单据号生成器，强制包含 `BATCH-`；<br>②历史错误条码（BC...）标记失效，按新规则重新生成；<br>③条码详情页标题显示完整复合条码，页面内展示关联物料主数据 | P0 |

### 3.2 仓储菜单与权限分配

| 角色 | 可见菜单 | 操作权限 | 说明 |
| :--- | :--- | :--- | :--- |
| **仓管员** | 仓储总览、入库单、出库单、库存查询、批次列表、库存预警、盘点单 | 增/改/删/导/打 | 全部仓储操作权限 |
| **生管** | 仓储总览、库存查询、库存预警 | 仅查看 | 只关注库存够不够排产，不操作入库 |
| **采购** | 仓储总览、入库单（仅采购到货部分）、库存预警 | 查看+催办 | 关注自己买的货到了没 |
| **品质** | 仓储总览、批次列表（仅查看质量状态） | 仅查看 | 质量追溯用 |
| **财务** | 仓储总览、库存查询（含成本） | 仅查看 | 库存金额核算 |
| **销售** | 无仓储菜单 | 无 | 销售不需要看库存细节 |

## 六、Mock 数据生成规范

> **背景**：当前系统截图中的演示数据（如 `WL-DEMO-STEEL-001`、`BC20260619-0002` 等）由 AI 随机生成，未遵循 PRD 编码规则，导致 UI 展示时出现逻辑矛盾（成品码绑原材料码、条码不含 BATCH 等）。为避免演示时给客户造成误解，**必须规范所有 Mock 数据的生成规则**。

| 序号 | 规范项 | 要求 | 示例 |
| :--- | :--- | :--- | :--- |
| **6.1** | **物料编码（料号）规则** | 前缀按物料分类区分：<br>• 成品/半成品：`WL-`<br>• 原材料：`RM-`<br>• 辅料/包材：`AU-`<br>后缀为字母数字组合（4-6位） | `WL-A001`（成品法兰盘）<br>`RM-STEEL-45`（45#圆钢）<br>`AU-GRIND-01`（砂轮） |
| **6.2** | **物料条码（条码号）规则** | 格式：`{物料编码}-BATCH-{YYYYMMDD}-{seq:4}`<br>必须包含物料编码+BATCH+日期+4位流水 | `WL-A001-BATCH-20260619-0001`<br>`RM-STEEL-45-BATCH-20260615-0003` |
| **6.3** | **图纸编号规则** | 格式：`DWG-{YYYYMMDD}-{seq:4}` | `DWG-20260619-0001` |
| **6.4** | **工单号规则** | 格式：`GD-{YYYYMMDD}-{seq:4}` | `GD-20260619-0001` |
| **6.5** | **委外单号规则** | 格式：`WW-{YYYYMMDD}-{seq:4}` | `WW-20260619-0001` |
| **6.6** | **采购订单号规则** | 格式：`PO-{YYYYMMDD}-{seq:4}` | `PO-20260619-0001` |
| **6.7** | **销售订单号规则** | 格式：`XS-{YYYYMMDD}-{seq:4}` | `XS-20260619-0001` |
| **6.8** | **检验单号规则** | 格式：`QI-{YYYYMMDD}-{seq:4}` | `QI-20260619-0001` |
| **6.9** | **条码与物料编码一致性校验** | **关键约束**：条码号中的 `{物料编码}` 部分必须与该条码关联的 `物料编码` 字段**完全一致** | ✅ 正确：条码号 `RM-STEEL-45-BATCH-...` 关联物料编码 `RM-STEEL-45`<br>❌ 错误：条码号 `WL-DEMO-STEEL-001` 关联物料编码 `RM-STEEL-45`（前缀不一致） |
| **6.10** | **Mock 数据生成脚本要求** | ①编写完整的数据填充脚本（`init-data.sql`），所有 Mock 数据必须遵循上述规则生成；<br>②每次重新部署 demo 环境时，运行该脚本清空并重新生成合规 Mock 数据；<br>③脚本需包含数据一致性自检（校验条码与物料编码前缀是否匹配） | — |
| **6.11** | **演示环境数据清理** | 当前所有不合规的 Mock 数据（`BC...`、`WL-DEMO-...` 等）需在下次演示前清理，按新规范重新生成 | — |

> **开发任务**：请在 `dev/scripts/` 目录下新建 `generate-mock-data.sql`，包含所有基础数据和业务数据的 INSERT 语句，并严格遵循上表编码规则。QA 在验收时需要检查条码与物料编码的一致性。
