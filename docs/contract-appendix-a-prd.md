## A.1 项目目标（G1~G10）

| 编号 | 目标 | 一句话描述 |
|------|------|-----------|
| **G1** | 报价到回款闭环 | 业务员从询价到回款全流程不切换系统；报价审批 <5万 业务员、5-20万 部门经理、>20万 总经理自动路由 |
| **G2** | 一码到底的车间协同 | 工单码、流转码、设备码贯穿开工→加工→报工→过站→入库，物料去向追溯到工序级 |
| **G3** | 排产与机台产能可视 | 生管员一屏看所有机台负荷、待排产工单、逾期预警，不依赖 Excel |
| **G4** | 委外灵活可控 | 自制/委外一键切换；工序级/整单级都支持；委外对账完成后才生成付款 |
| **G5** | 数据一次录入多端共用 | 业务单据录入一次自动同步到财务、车间报表、老板驾驶舱 |
| **G6** | 一线人员零培训上手 | 操作工扫码开工到报工 3 步搞定；业务员录报价到转订单 5 分钟完成 |
| **G7** | 数据权限隔离 | 管理层见成本/毛利/利润率；一线只见数量/工序/状态 |
| **G8** | 供应商一等公民 | 本地部署，厂外不开放系统账号，163 邮箱 + 纸质签字盖章 + 拍照上传 |
| **G9** | 委外端到端可追溯 | 7 状态机 + 返修次数累加 + ≥2 次自动预警高层，一码留痕 |
| **G10** | 料号维度数据洞察 | 搜一个料号，看到该料号最近 12 个月价格、5 段成本结构 |

## A.4 灵魂设计：扫码三码体系 + 委外单码

> **V1.3.5 重大变更**：V1.3.4 引入的"区域码 AREA-"整章删除，扫码类型由 4 类降为 3 类 + 委外单码。

| 码 | 命名规则 | 一码承载信息 | 触发场景 |
|----|---------|------------|---------|
| **GD-** 工单码 | GD-YYYYMMDD-NNNN | 工单号/产品/工序/数量 | 扫码开工、报工、查工单 |
| **LZ-** 流转码 | LZ-GDxxx-NN（工序委外时为 LZ-GDxxx-P{工序号}） | 流转ID/工单/工序/合格数量/下工序 | 工序交接、过站、转工 |
| **SB-** 设备码 | SB-CCC-NNN | 设备ID/名称/机台号 | 扫码选机台、产能查询 |
| **WL-** 物料码 | 物料编码+批次+入库日期+流水 | 物料编码/批次/库位 | 扫码出入库 |
| **WW-** 委外订单码 | WW-YYYYMMDD-NNNN | 委外单号/料号/数量/约定交期 | 仓管到货扫码 |

> **设计承诺**：新人 5 分钟通过"扫三码"读懂"工单是什么 → 现在到哪道工序 → 在哪台设备做"。

## A.5 13 个 Epic 详情（60+ Story / 200+ AC 摘要）

### Epic 1 基础设施与权限（4 Story）
- 用户与角色权限、审批工作流配置、系统参数与数据字典、APP 端基础
- 金额权限：业务员、经理、总经理三类内置角色，阈值在角色管理可调
- 审批超时：待办超 24h 系统消息 + 邮件提醒

### Epic 2 客户与销售（4 Story）
- 客户档案与 CRM、报价与多级审批、订单与合同、合同回款与利润
- 客户保护期默认 30 天可配
- 报价/订单审批阈值同 G1

### Epic 3 图纸与物料（4 Story）
- 图纸与版本管理、工程转化、BOM 多级维护、工艺库与工序
- 物料编码规则：`YL-45-D80` 表示圆料 45#钢 直径 80mm
- BOM 支持不限层级，损耗系数按层级独立维护

### Epic 4 扫码仓储（4 Story）
- 物料条码生成、APP 扫码出入库、库位批次与多仓库、安全库存与预警
- 库位编码：`W-A-01-03` 表示一号仓库 A 区 01 排 03 位
- 实时库存 = 在库 + 在途 - 冻结

### Epic 5 生产执行核心（4 Story）
- 工单与排产、扫码开工/报工/过站、委外下单基础、设备与机台排产
- 工单状态机：草稿→待排产→已排产→加工中→已报工→已检验→已入库→已关闭（7 主状态）
- **V1.3.7 工序分配职责严格分离**：生管只决定"哪些工序自制/委外"，不指定厂商；采购只选厂商，不决定自/委

### Epic 6 委外加工深化（7 Story）
- 委外对账与付款、工序/整单委外灵活切换、自动调取委外历史价、委外质检、**7 状态机**、**返修闭环 + 次数预警**、**委外历史交期预估**
- 委外历史价：自动带出该厂商该工序最近 3 次成交价（取中位数）
- 返修次数阈值：默认 2 次，Nacos `app.outsub.rework-alert-threshold` 热更新
- 历史交期预估：`estimated_completion = order_date + median(recent_3_orders.delivery_days)`

### Epic 7 品质管控（4 Story）
- 来料/过程/成品检验、FA 首件、三次元检测、不良品处理
- FA 双签：品检 + 工程师
- 不良原因分类：材料/工艺/设备/人为

### Epic 8 采购管理（4 Story）
- 询价比价、价格控制、到货提醒、来料质检
- 采购审批阈值：<1万 采购员、1-5万 采购主管、>5万 总经理
- 限价规则：物料维度配置建议进价上限

### Epic 9 财务与成本（5 Story）
- 应收应付、成本核算、回款控制、利润分析、**料号成本聚合视图**
- 利润率分级预警：毛利率 <10% 标黄、<5% 标红、<0 深红 + 老板驾驶舱强提示
- 料号 5 Tab：价格 / 材料成本 / 工时成本 / 外协成本 / 总成本
- 付款触发三条件：对账已确认 + 签字扫描件已上传 + 双方对账金额一致

### Epic 10 人事管理（3 Story）
- 员工档案与考勤、薪酬自动核算、绩效与招聘
- APP 打卡（WiFi/蓝牙定位）+ 请假/调休/加班申请
- 工资账套：基本 + 岗位 + 绩效 + 加班 + 全勤 - 社保 - 个税

### Epic 11 报表与看板（5 Story）

## A.7 委外订单 7 状态机详图（PRD §附录 B-2）

```
PENDING_SHIP（待发货）
  └─ 厂商发货 + 仓管代录 → SHIPPING（送货中）
       └─ 仓管 APP 扫 WW- 委外单码 → PENDING_INSPECTION（待检）
            └─ 品质员去仓库领料 → INSPECTING（质检中）
                 ├─ 合格 → QUALIFIED_STORAGE（待入库）
                 │    └─ 仓管扫物料码 + 录库位 → STORED（已入库） [终态]
                 │         └─ 触发工序接续 E5-S6 + 对账归集 E6-1
                 └─ 不合格 → REPAIR_REQUESTED（待返修）
                      └─ 采购生成返修单 → NOTIFIED_REPAIR（已通知返修）
                           └─ 厂商再送货 → 回到 SHIPPING（is_rework_reinspection=true）
                                └─ 返修次数 ≥ 2 → 自动预警高层（APP + PC 红点 + 企业微信 webhook）
```

**触发器与通知规则（V1.3.7 改版：通知厂商统一改为 163 邮箱单渠道）**

| From | To | Trigger | 通知对象 |
|------|----|---|---------|
| PENDING_SHIP | SHIPPING | 厂商电话/系统外接单 + 仓管代录 | **厂商：163 邮箱**（含 PDF 委外订单详情） |
| SHIPPING | PENDING_INSPECTION | 仓管 APP "到货扫码"扫 WW- | **生管 + 品质**（APP + 企业微信 + PC 红点） |
| PENDING_INSPECTION | INSPECTING | 品质员去仓库领料 | 生管（CC 通知） |
| INSPECTING | QUALIFIED_STORAGE | 品管录入"合格" | 仓管（APP） |
| INSPECTING | REPAIR_REQUESTED | 品管录入"不合格 + 退回维修" | 采购（APP + 企业微信）+ **厂商：163 邮箱** |
| QUALIFIED_STORAGE | STORED | 仓管扫物料码 WL- + 录库位 | 生管（工序接续）+ 财务（对账归集） |
| REPAIR_REQUESTED | NOTIFIED_REPAIR | 采购生成返修单 | **厂商：163 邮箱** + 生管 |
| NOTIFIED_REPAIR | SHIPPING | 厂商再送货 | 仓管（待下次扫码） |
| 任意 → 返修次数 ≥ 2 | 任意 | 自动触发 | **高层 + 采购主管 + 生管主管**（APP + PC 红点 + 企业微信 webhook） |

## A.8 月度对账状态机（V1.3.6 改版 / V1.3.7 简化，PRD §附录 B-3）

```
DRAFT → SENT（采购邮件发对账单 PDF）
  └─ 厂商线下流程不在 PRD 描述 → CONFIRMED（采购上传签字扫描件）
       └─ 财务校验双方金额一致 → PAYING（生成付款申请 + 走双签流程）
            └─ 财务付款 → PAID（终态）/ REJECTED（异常：金额不一致 / 附件缺失 → 退回重对）
```

**签字扫描件安全**：
- AES-256-GCM 加密 + MinIO 存储
- 5 年保留
- 下载权限：总经理 / 财务总监 / 采购员
- 预览/下载全审计
