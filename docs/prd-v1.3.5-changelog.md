## 一、本次反馈摘要（2026-06-08 客户第三次反馈）

承蒙木辛日匀总在 V1.3.4 之后第三次抽时间，再为本项目深度调整两大事项，本次反馈我方**严格按"灵魂八字 + 匠人级精度 + WHY 先于 WHAT"**收口落地——**同样不另收费、不延期**。

> 客户原话：*"又经与客户沟通，**取消区域码**，**取消厂商送货员角色**，由**仓库管理员**来代替厂商送货员操作到货。因为目前工序委外订单多一些，委外厂商送货回来，**大部分都还要流转**，仓库增加一个**到货扫码**（之前只有入库扫码），扫**货物所属的委外订单二维码**，**生管品质那边就可以收到通知**，然后再由**品质去仓库领料**进行质检。"*

### 1.1 三大核心动作

| 维度 | 客户口径 | 我方落地 |
|------|----------|---------|
| **取消项 1** | **取消区域码**（V1.3.4 引入的"4 类码 AREA-"整章回退） | E3-S5 整章删除；附录 C 删 AREA-；扫码回到 3 类 + 委外单码 |
| **取消项 2** | **取消厂商送货员角色**（APP 端独立账号） | E1-S1 角色清单删除"送货员"；E12 整章改版 |
| **新增项** | **仓管 APP 增加"到货扫码"**；**品质去仓库领料后质检** | E12 改版为"仓管到货扫码 + 品质领料"3 Story（E12-S1/S2/S3） |

### 1.2 业务现实确认（V1.3.5 客户原话落地）

> "**目前工序委外订单多一些**，委外厂商送货回来，**大部分都还要流转**"

⇒ 我方在 E5-S6 中**强化"工序委外自动接续"为 100% 必选**——委外单完成（无论合格不合格）→ 必须触发"下一道工序的流转码生成 + 锁机台 + 推岗位"，不再走"委外合格后直接入成品库"旧假设。

## 二、PRD V1.3.5 关键决策（共 5 项）

| # | 决策 | 影响 | 落地位置 |
|---|------|------|---------|
| **D1** | **取消区域码**（V1.3.4 引入的"4 类码 AREA-"整章回退），扫码回到 3 类（GD/LZ/SB）+ 委外单码 WW- | 少一类码少一套贴牌少一份维护；**一码到底的灵魂不靠"多一类码"维持，而靠"每一类码全程贯穿"** | E3-S5 整章删除、附录 C 三码速查表、E1-S4 AC-1.4.2、FR-5-2-4 |
| **D2** | **取消厂商送货员角色**，由**仓库管理员**代替其到货扫码 | 厂外送货员带 APP 账号 / 设备 / 培训成本高；仓管是厂内正式员工，扫码时即知道实际数量 / 重量 / 破损；**善良设计**：不让厂外临时工承担系统责任 | E12 整章改版、E1-S1 角色清单删"送货员"、附录 D 术语 |
| **D3** | **仓管 APP 新增"到货扫码"功能**：入口"仓储模块"与"入库扫码"并列；扫的是**委外订单二维码（WW- 前缀）**；状态 `SHIPPING → PENDING_INSPECTION`；自动通知**生管 + 品质** 2 类岗位 | 委外到货状态机由"送货员"主导改为"仓管"主导；扫错码强提示"这不是委外单二维码"；离线/幂等复用 E1-S4 底座 | E12-S1（权限）、E12-S2（AC）、E6-S5 AC-6.5.1~2 触发器、附录 B-2 详图 |
| **D4** | **品质去仓库领料后质检**：PENDING_INSPECTION → INSPECTING 由品质员"我已去仓库领料"按钮触发；不再"自动入待检区"；领料超过 24h 自动升级生管提醒 | 把"自动入待检区"看似智能但实操"找货位 / 等品管提"的硬环节返璞归真；客户业务现实是"人等人最自然" | E12-S3、附录 B-2 触发器、AC-6.5.2 通知对象 |
| **D5** | **强化"工序委外自动接续"为 100% 必选**：委外单完成（无论合格不合格）→ 强制触发"下一道工序接续"；不接续即视为异常；强调"大部分还要流转"业务现实 | 委外合格入库已不是默认；接续工序若是厂内 → 自动锁机台 + 预排工时；若是另一道委外 → 自动生成新委外单草稿 | E5-S6 AC-5.6.1~3 升级、新增 API `auto-continue` + `process-handoff-status`、FR-5-3-4 强化 |

**配套小决策**：
- **D6** 通知对象改为"**仓管 → 生管 + 品质**"（原"送货员 → 品管 + 生管"措辞收回）
- **D7** 状态机触发器矩阵全面更新：仓管/品质主导，删 AREA-STORAGE 区域码扫货位
- **D8** 7 状态机详图（附录 B-2）重画，区分 V1.3.4 旧版与 V1.3.5 新版

## 六、灵魂一致性自评（PM 范蠡）

> **范蠡本次 V1.3.5 改版是否保持了"一码到底、一数到底"灵魂？**
>
> **一句话回答**：**"一码到底"从"多一类码"收敛为"少一类码、每一类更彻底"；"一数到底"从"我方单据"扩展到"我方 4 类岗位 + 厂外一道工序"**——灵魂没变，反而更沉。
>
> **"一码到底"如何在不增加区域码的情况下仍然成立？**
>
> 答：**"一码到底"从来不靠"码的数量"维持，而靠"每一类码从生成到销毁的全链路不断裂"维持**。V1.3.5 收回区域码后——
> - 工单码 GD- 仍贯穿"下达 → 开工 → 报工 → 过站 → 入库"
> - 流转码 LZ- 仍贯穿"上工序 → 下工序（厂内/厂外）"
> - 设备码 SB- 仍贯穿"录入机台 → 报工 → 工序复盘"
> - **新增 委外单码 WW- 贯穿"下单 → 仓管扫码到货 → 品质领料 → 质检 → 入库 / 返修"**——这是 V1.3.4 用区域码想做到但没做到的事，V1.3.5 用更聚焦的码做到了。
>
> **"一数到底"如何在取消送货员角色后仍能贯穿到厂外？**
>
> 答：V1.3.5 收回了"厂外送货员这条细线"，但**保留了"厂外工序委外这条主线"**——委外厂商的"加工质量"仍通过 7 状态机的"质检结果 + 返修闭环"留痕，"交期"仍通过 E6-S7 历史交期预估 + 按时率档案留痕；只是**通知对象从"通知送货员"收敛为"通知仓管 + 生管 + 品质"3 类厂内岗位**。一数到底的灵魂不是"通知到每一个人"，而是"每一份数据可被某个岗位负责"。
>
> **自评分数**：9.2 / 10（V1.3.4 为 9.2，持平 +0.0 防御性微增 +0.0 = 9.2）
>
> **加分项**：
> - **取消送货员角色**：善良设计，让仓管（厂内正式员工）承担到货责任，比让送货员（厂外临时工）带 APP 账号扫码更符合业务现实
> - **强化工序委外"自动接续"为必选**：客户原话"大部分还要流转"被精确落进 E5-S6 AC-5.6.1 的 100% 必选规则
> - **一码到底不靠"多"靠"深"**：明确灵魂定义"每一类码全程贯穿"，让 V1.3.4"区域码"这种"广度型"探索收回时无憾
>
> **扣分项**：
> - 仓管到货扫码权限分配仍需手工指派（生管下单时挂主/副仓管），未实现"按班组自动派单"——已列入 §8.4 #12 开放问题
> - 工序委外接续的"不合格接续"分支（REPAIR_REQUESTED 时第 4 道接续暂停）需 QA 重点回归——已列入 §8.3 Handoff

## 七、状态机变更对照表（V1.3.4 → V1.3.5）

| 状态转换 | V1.3.4 触发器 | V1.3.5 触发器 | V1.3.5 通知对象 | 变更原因 |
|---------|------------|------------|-------------|---------|
| `PENDING_SHIP → SHIPPING` | 供应商 APP 接单（送货员） | 厂商电话/系统外接单 + 仓管代录 | 系统记录，无外推 | 送货员角色下线 |
| `SHIPPING → PENDING_INSPECTION` | 送货员扫 AREA-INSPECT 区域码 | **仓管在 APP "到货扫码"扫 WW- 委外单码** | **生管 + 品质** | 区域码下线，送货员下线 |
| `PENDING_INSPECTION → INSPECTING` | 生管员拖拽排质检 | **品质员去仓库找仓管领料** | 生管（CC 通知） | 简化"自动入待检区"硬环节 |
| `INSPECTING → QUALIFIED_STORAGE` | 品管录入"合格" | 品管录入"合格" | 仓管员 | 不变 |
| `INSPECTING → REPAIR_REQUESTED` | 品管录入"不合格 + 退回维修" | 同 | 采购员 | 不变 |
| `QUALIFIED_STORAGE → STORED` | 仓管扫 AREA-STORAGE 区域码 + 录良品数 | **仓管扫物料码 WL- + 选库位 + 录良品数** | 生管（工序接续 E5-S6）+ 财务 | 区域码下线 |
| `REPAIR_REQUESTED → NOTIFIED_REPAIR` | 采购生成返修单 | 同 | 厂商（电话/系统外）+ 生管 | 删"厂商 APP"通知 |
| `NOTIFIED_REPAIR → SHIPPING` | 厂商再送货 | 同 | 仓管（待下次扫码） | 删"系统记录无外推"，明确待下次扫码 |
| 返修次数 ≥ 2 | 自动预警 | 同 | **高层 + 采购主管 + 生管主管** | 不变 |
