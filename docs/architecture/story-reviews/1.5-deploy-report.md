## 关键反馈（已闭环 · 部署阶段纳入）

1. **V1.3.7 6 类越权扫描 100% 脚本化**：ZAP 脚本覆盖 IDOR / 状态机 / 金额 / 候选人 / 审计 / 二级密码（V1.3.7 红线 1/2/5/6 全覆盖）
2. **3 性能基线阈值化**：路由 P95<50ms / 转订单 P95<1s / PDF P95<2s 全部硬编码到 k6 `thresholds`，CI 跑过即合规
3. **6 E2E 4 角色项目化**：salesperson / dept-manager / gm / finance 4 个 Playwright project，per-role 登录 fixture，testcontainers mysql 共享
4. **15 跨模块契约测例 100% 覆盖 8 个下游 Epic + Story 1.4**：覆盖 quote → order / 委外 / 品质 / 采购 / 财务 / 报表 / APP 5 类码 7 大下游
5. **docker-compose 启动顺序严格化**：mysql-master (health) → nacos (health) → erp-platform (health) → erp-business (health)，5 分钟超时 + 启动期 60~180s 容忍 Spring Boot 冷启
6. **辅助修复 erp-platform audit/annotation 包**：原项目 erp-platform 缺 `com.btsheng.erp.platform.audit.annotation` 子包导致 erp-business 编译失败，补 `@interface AuditLog` 转发注解解决

## 关注点（不阻断 · 部署实测阶段）

1. **k6 / Playwright / ZAP 实际跑需部署环境**：本机无 k6 命令 + 无 Playwright 浏览器 + 无 ZAP Docker 镜像，3 脚本 + 6 E2E + 1 ZAP 留 CI 部署环境实测
2. **docker-compose 启动慢**：erp-business 健康检查 start_period=180s，整个 stack 启动需 5-8 分钟（Spring Boot 冷启 + V4 迁移）
3. **testcontainers MySQL 8 镜像拉取慢**：web-impl E2E 首次跑需拉 mysql:8.0 镜像（约 500MB），CI 加 cache 优化
