# 3. 用户界面设计目标

### 3.1 整体 UX 愿景

**桌面端（Web）**：信息密度高、表格为主、键盘流为主。业务员/生管/财务每天操作 100+ 单据，效率优先。**核心范式**：列表 + 详情 + 表单三件套，命令面板（Ctrl+K）直达功能。

**移动端（Android APP）**：操作工/仓管戴手套、车间光线差、站着操作。**核心范式**：扫码即一切。APP 首页 80% 区域是"扫码框"；剩余 20% 是"待办 + 消息"。

**傻瓜测试**：操作工奶奶能学会，那 UI 就对了。例：扫码报工只需 3 步——扫工单码 → 输数字 → 按"提交"，无任何菜单。

### 3.2 核心交互范式

| 场景 | 范式 | 设计要点 |
|------|------|----------|
| 桌面端列表 | 高密度表格 + 多条件过滤 + 快捷搜索 | 行高 32px，斑马纹，列宽可调 |
| 桌面端表单 | 分步骤/折叠面板 | 长表单折叠；金额输入右侧计算器 |
| 移动端扫码 | 全屏扫码 + 大按钮 + 语音/震动反馈 | 扫描成功震动 + "滴"声；按钮 ≥ 64px |
| 审批 | 待办列表 + 一键通过/驳回 | 移动端支持滑动手势 |
| 看板 | 卡片拖拽 + 实时刷新（WebSocket 5s） | 关键看板支持全屏 |
| 报表 | 图表（折线/柱状/饼图）+ 表格下钻 | 时间范围默认近 30 天 |

### 3.4 委外加工管理 · 工序分配职责（V1.3.7 新增）

> **工序分配职责（V1.3.7 新增）**——工序的"自/委"决定与"厂商选择"必须**严格分离**，由生管与采购各司其职：
>
> | 决策内容 | 责任角色 | 在哪里 | 怎么操作 |
> |---------|---------|-------|---------|
> | 工序划分（厂内 / 委外） | **生管** | PC 端工单详情 | 看订单工序清单 → 勾选每道工序的归属（厂内/委外）|
> | 厂商选择（哪家厂商） | **采购** | PC 端委外管理 → 待委外清单 | 接收生管提交的待委外工序列表 → 选厂商 + 填价 |
>
> **流程**：生管完成工单工序划分 → 系统生成「待委外工序清单」→ 推送给采购（APP + PC 端红点 + 站内信）→ 采购为每道工序选厂商并创建委外单（WW- 单号 + 二维码）→ 自动 163 邮箱通知厂商。
>
> **业务意义**：
> - 工序划分 = 生产计划问题（生管懂生产排程、机台负荷、交期）
> - 厂商选择 = 供应链问题（采购懂价格、谈判、厂商能力、付款）
> - 职责清晰 = 避免"生管指定某厂商但采购拿不到好价格"或"采购选了厂商但生管没安排工序"的脱节
> - 善良设计：不越权、不甩锅，每个角色只做自己擅长的事
>
> **灵魂一致性**："一数到底"不靠"一个角色包打天下"维持，而靠"每个角色只对自己的数据负责"维持。**V1.3.7 把"工序分配"和"厂商选择"严格分离**（详见 §2.1 FR-5-3 工序分配职责段、E5-S4 改版、E6-S8 厂商资料字段调整、附录 D 术语表）。

### 3.5 目标设备与平台

| 端 | 设备 | 操作系统 | 浏览器/版本 |
|----|------|----------|------------|
| Web | 桌面 PC / 笔记本 | Windows 10+ / macOS 12+ | Chrome / Edge / Firefox 最新 |
| Android | 手机 / PDA（霍尼韦尔、斑马） | Android 8.0+ | 原生 APP（Kotlin） |
| iOS | V1.0 不支持 | - | - |

> **iOS 决策**：V1.0 不发布 iOS 端。原因：CNC 行业一线工人 iOS 设备渗透率 < 5%；Android 端覆盖 95% 场景；iOS 端 V1.1+ 评估。
