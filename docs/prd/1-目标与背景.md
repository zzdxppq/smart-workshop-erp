# 1. 目标与背景

### 1.1 项目目标

- **G1. 报价到回款闭环**：业务员从询价到回款全流程不切换系统；报价审批<5万业务员、5-20万部门经理、>20万总经理自动路由。
- **G2. 一码到底的车间协同**：工单码、流转码、设备码贯穿"开工→加工→报工→过站→入库"，物料去向可追溯到工序级。
- **G3. 排产与机台产能可视**：生管员在一个看板上看见所有机台负荷、待排产工单、逾期预警；不依赖 Excel。
- **G4. 委外灵活可控**：自制与委外可一键切换，工序级/整单级委外都支持；**V1.3.6** 委外对账完成后 + 厂商签字扫描件上传 + 双方金额一致 才一键生成付款申请。
- **G5. 数据一次录入多端共用**：业务单据录入一次，自动同步到财务应收应付、车间报表、老板驾驶舱，杜绝"两个数"。
- **G6. 一线人员零培训上手**：操作工扫码开工到报工，3 步搞定；业务员录入报价到转订单，5 分钟内完成。
- **G7. 数据权限隔离**：管理层可见成本/毛利/利润率；一线（操作工/仓管/品检）只见数量/工序/状态/关联图纸，不见金额。客户与供应商通过 APP 协同时只能看到与自己相关的数据。（V1.3.9 Sprint 12 Story 12.1 更新：操作工可见关联图纸）
- **G8. 供应商一等公民**（V1.3.3 追加 / V1.3.5 收回 APP 账号 / **V1.3.6 改版为"邮件 + 短信 + 线下签字"** / **V1.3.7 改版为"单一 163 邮箱 + 工序分配职责分离"**）：本地部署场景下，**厂外不开放系统账号**，改为"**163 邮箱推送 + 月度对账单 PDF 邮件 + 厂商签字盖章后采购拍照上传**"；**V1.3.7 进一步把"工序分配（生管）"与"厂商选择（采购）"严格分离**——把对账的法律凭证沉淀进系统，同时不增加厂外任何 IT 负担。
- **G9. 委外端到端可追溯**（V1.3.4 追加 / V1.3.5 收回区域码）：委外订单从下单 → 送货中 → 待检 → 质检中 → 合格入库 / 退货维修 → 已通知返修 → 返修复检 → 已对账 → 已付款，**七主状态 + 返修次数累加 + ≥2 次自动预警高层**，**V1.3.5 改为由仓管扫"委外订单二维码"触发到货（不再走区域码）**，全程一码留痕，无任何"线下电话/微信"环节遗失。
- **G10. 料号维度数据洞察**（V1.3.4 追加）：从单笔订单 → 单料号生命周期，业务员 / 工程师 / 财务 / 管理层在 PC + APP 任意端**搜一个料号**，即可看到该料号最近 12 个月价格、材料/工时/表处/外协/管理费分摊完整成本结构，让老板秒看"哪类料赚不赚"。

### 1.2 项目背景

#### 1.2.1 行业与痛点

CNC 机加工行业有八大鲜明特征：非标定制件多、工序流转复杂（车床→CNC→后加工→表处→质检→入库）、委外加工频繁（氧化/电镀/喷涂/热处理/三次元）、图纸是核心资产、编程管理依赖历史积累、材料按料号+尺寸追踪损耗、产能依赖设备状态、品质检测严苛（三次元/FA 首件）。通用 ERP 解决不了这些细分场景——车间里老师傅的经验在脑子里，新人来了要跟一年；订单交期靠 Excel，逾期了老板才知道；**委外对账**（V1.3.6 客户原话）"因为是**本地部署**，所以厂商没有微信操作这一部"——通知改邮件/短信，对账靠采购**线下拿对账单 + 厂商签字盖章 + 拍照上传**，对账的法律凭证沉淀进系统。

#### 1.2.2 解决路径

本系统走"行业 ERP + 匠人级扫码 + 一数到底"路线。前端 Web 端面向业务员、工程师、生管、仓管、品检、财务、人事等桌面办公角色；Android APP 端面向一线操作工（车间）和仓管（仓库），聚焦扫码开工/报工/过站/出入库。技术上采用 Spring Cloud Alibaba 微服务（≤3 个后端服务 + 1 个 backend-internal `core` 公共 Module，与微服务同级放在 backend 仓内），polyrepo 三仓（backend / web / android）管理，避免单仓膨胀与仓间版本同步成本。消息中间件使用 **Redis Stream + XXL-JOB** 替代传统 MQ——CNC 单机部署场景下，Redis 已存在，Stream 的消费者组能力足够覆盖消息通知、扫码同步、轻量异步任务；定时任务由 XXL-JOB 统一调度。
