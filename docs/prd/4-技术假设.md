# 4. 技术假设

### 4.1 仓库结构（Polyrepo，三仓）

> **为什么 polyrepo 三仓？** 微服务 + 多端的最佳实践。每个端独立版本号、独立发布、独立失败域。`core` 公共依赖放在 **backend 仓内作为 Maven Module**（与 `erp-business` / `erp-production` / `erp-platform` 同级），而非独立成仓——避免"改一行 DTO 跨仓发包"的版本同步成本，仍能复用 80% 通用代码。

```
smart-workshop-erp-backend/   ← 后端整体（含 core Module + 3 个微服务 Module）
   ├─ core/                    ← 公共依赖（common-dto / common-util / common-entity / common-web）
   ├─ erp-business/            ← 业务服务（Spring Boot Application）
   ├─ erp-production/          ← 生产服务
   ├─ erp-platform/            ← 平台服务
   └─ erp-gateway/             ← Spring Cloud Gateway

smart-workshop-erp-web/        ← Vue 3 + TypeScript + Vite + Element Plus
smart-workshop-erp-android/    ← Kotlin + Jetpack（V1.1 Compose）
```

**Maven 仓内模块依赖**（backend 单仓多 Module）：

```
erp-gateway ──┐
              ├─→ core
erp-business ─┤
erp-production┤
erp-platform ─┘
```

**跨仓依赖**：
- `web` 不直接依赖 backend 内部 `core`；通过 OpenAPI 自动生成 TypeScript 客户端。
- `android` 不直接依赖 backend 内部 `core`；通过 OpenAPI 自动生成 Kotlin 客户端。
- 三个仓的版本号独立演进（`backend` 用 `maven-release-plugin` 管 internal 版本；`web`/`android` 用语义化版本）。

### 4.2 服务架构

> **为什么 ≤ 3 个业务服务？** 业务初期，过度拆分会增加运维复杂度和分布式事务问题。"粗粒度微服务 + 仓内模块化单体"是务实选择。

```
                         ┌────────────────────────────┐
                         │  Nginx (Web/APP 静态 + 负载均衡) │
                         └─────────────┬──────────────┘
                                       ↓
              ┌──────────────────────────────────────────┐
              │  erp-gateway (Spring Cloud Gateway)       │
              │  - JWT 鉴权  - 限流  - 路由  - 日志        │
              └─────────────┬────────────────────────────┘
                            ↓
       ┌──────────────────┬──────────────────┬──────────────────┐
       │ erp-business     │ erp-production   │ erp-platform     │
       │ (业务聚合)        │ (车间执行)        │ (基础能力)        │
       │                  │                  │                  │
       │ 客户服务          │ 工单服务          │ 用户服务          │
       │ 报价服务          │ 工序服务          │ 角色权限服务      │
       │ 订单服务          │ 扫码服务          │ 审批流服务        │
       │ 合同服务          │ 报工服务          │ 系统参数服务      │
       │ 采购服务          │ 委外服务          │ 字典服务          │
       │ 仓储服务          │ 设备服务          │ 文件服务          │
       │ 品质服务          │ MRP 服务          │ 消息服务          │
       │ 财务服务          │                  │                  │
       │ 人事服务          │                  │                  │
       │ 报表服务          │                  │                  │
       └────────┬─────────┴────────┬─────────┴────────┬─────────┘
                ↓                  ↓                  ↓
                └──────────────────┬──────────────────┘
                                   ↓
                    ┌──────────────────────────────┐
                    │  MySQL 8.0  +  Redis 7       │
                    │  Redis Stream (消息/事件)     │
                    │  XXL-JOB  (分布式调度)        │
                    │  MinIO     (图纸/文件)        │
                    └──────────────────────────────┘
```

> **服务拆分原则**：`erp-business` 装所有业务 CRUD（轻聚合，重数据库），`erp-production` 装车间执行（重实时性，独立部署），`erp-platform` 装基础能力（被前两者依赖，禁止反向依赖）。**单服务多 Module（Modulith）** 而非无限拆服务；`core` 公共 Module 在 backend 仓内为所有服务共享。

### 4.2.1 消息与异步任务方案（Redis Stream + XXL-JOB）

> **为什么不用 RabbitMQ？** CNC 单机部署场景下，引入 RabbitMQ 额外多一个进程、多一份内存、多一套运维。Redis 在系统中已存在（缓存/Session/分布式锁），其 **Stream 数据结构** 提供消费者组、ACK、Pending 重投等能力，足以覆盖本系统全部消息场景；**XXL-JOB** 接管所有定时/周期任务。两者组合，少一个中间件、少一份故障域。

| 场景 | 原方案（RabbitMQ） | 新方案（Redis Stream + XXL-JOB） |
|------|--------------------|----------------------------------|
| 消息通知（审批/逾期/异常） | RabbitMQ Topic | Redis Stream `stream:notify` + 消费者组 |
| 扫码数据异步同步 | RabbitMQ Direct | Redis Stream `stream:scan-sync` + 消费者组 |
| 报表统计异步计算 | RabbitMQ + Cron | XXL-JOB Cron 任务 + Redis 缓存结果 |
| 回款到期提醒 | RabbitMQ 延迟队列 | XXL-JOB 定时扫描 + 提醒发送 |
| 库存预警 | RabbitMQ 延迟队列 | XXL-JOB 每 10 分钟扫描 |
| MRP 重算 | RabbitMQ 触发 | XXL-JOB 手动触发 + Redis Stream 广播 |
| 分布式事务（业务→财务） | 事务消息 | 本地消息表 + XXL-JOB 补偿扫描 |

**Redis Stream 消费者组示例**：
- `stream:notify` ← `notify-consumer-group` ← [web-push-consumer, sms-consumer, workwechat-consumer]
- `stream:scan-sync` ← `scan-consumer-group` ← [business-consumer]

**幂等性保障**：所有消费者实现幂等键（业务单号 + 事件 ID），重复消费自动去重（Redis SET NX EX 缓存）。

### 4.3 测试要求

**测试金字塔**：

```
            /\
           /E2E\          Playwright（Web 关键路径 10 个）；Appium（APP 扫码 3 个）
          /------\
         /  集成  \        Testcontainers（MySQL/Redis 真实依赖）
        /----------\
       /   单元测试  \      JUnit 5 + Mockito + AssertJ（覆盖率 ≥ 70%）
      /--------------\
```

- **单元测试**：每个 Service / 关键 Util；覆盖率 ≥ 70%；金额计算/状态机 ≥ 90%。
- **集成测试**：Testcontainers 启动真实 MySQL/Redis；测试 Service 协作。
- **契约测试**：Spring Cloud Contract；后端与前端 OpenAPI 双向校验。
- **E2E**：Web 用 Playwright；APP 用 Appium 模拟扫码（mock 摄像头）。
- **性能测试**：JMeter 模拟 100 用户并发，扫码 50 并发。

### 4.5 其他技术假设

- **注册中心与配置中心**：**Nacos 2.3+**（Spring Cloud Alibaba 标配组件，承担服务注册发现 + 动态配置 + 命名空间隔离）。`erp-gateway` / `erp-business` / `erp-production` / `erp-platform` 全部向 Nacos 注册；各服务的 Nacos 配置按 `dev` / `staging` / `prod` 命名空间隔离，支持热更新（DB 凭证、限流阈值、扫码频率等可在线调整无需重启）。
- **数据库**：MySQL 8.0（主从 + 读写分离）；Redis 7（缓存 + Session + 分布式锁）。
- **消息队列**：**Redis Stream**（消费者组：消息通知/扫码同步/轻量异步任务）+ **XXL-JOB 2.4**（分布式调度：定时/周期任务）。**不引入 RabbitMQ**——CNC 单机部署场景下 Redis 已存在，Stream 能力足够。
- **文件存储**：MinIO（私有化 S3 兼容，图纸/合同/检测报告）。
- **定时任务**：XXL-JOB 2.4（分布式调度，回款提醒/库存预警/MRP 重算）。
- **认证**：JWT（access 2h + refresh 7d）+ Redis 黑名单。
- **API 网关**：Spring Cloud Gateway（限流/路由/认证/日志）。
- **监控**：Prometheus + Grafana；告警走企业微信。
- **CI/CD**：GitHub Actions；镜像推送阿里云 ACR。
- **开发规范**：阿里巴巴 Java 手册 + ESLint + Prettier + ktlint。
- **【V1.3.3 部署演进路径】**：
  - **阶段 1（V1.0 上线）**：CNC 工厂内部 8C32G 单机 + Docker Compose，所有用户走局域网 WiFi。供应商进厂连 WiFi 可用 E12 协同。
  - **阶段 2（V1.1 评审中）**：客户调研运营商公网 IP / 阿里云费用（客户待办）。决定后引入 HTTPS 证书 + Nginx 反向代理 + WAF + JWT 缩短 + 异地登录二次验证，将 E12 供应商协同开放给厂外。
  - **阶段 3（V1.2+）**：业务量增长后可平滑迁移阿里云 ECS 集群（Nacos 多节点 + MySQL 主从），扩至千人并发，无需改动业务代码。
