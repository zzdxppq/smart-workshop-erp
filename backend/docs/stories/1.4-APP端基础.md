## Epic Context

- **WHY（来自 PRD）**：客户原话"扫一扫就能到货、扫一扫就能报工"—— APP 是仓管/生管/品管/业务员"24 小时在线"的工具，必须能扫码、能离线、能推送。V1.0 仅做"基础壳"（登录 + 消息 + 扫码 + 离线缓存），具体业务功能（扫码到货、扫码报工、扫码过站）留给 Epic 4-仓储 / Epic 5-生产。
- **业务红线**（V1.3.7）：
  ① 账号密码登录 + 记住密码 + 指纹解锁（V1.0）/ 生物识别（V1.1，留 hook）
  ② 消息中心：审批通知（Story 1.2 4 通道推送之一）+ 逾期提醒（Story 1.2 `stream:notify`）+ 异常上报（仓管/品管扫码异常）+ 扫码记录回执
  ③ 扫码壳：5 类码统一识别（**GD-工单 / WL-物料 / LZ-流转 / SB-设备 / WW-委外**），按前缀自动路由
  ④ 离线模式：本地 SQLite（Room ORM）+ 冲突解决（以服务端最新状态为准，弹窗提示用户选择"覆盖"或"合并"）
- **解耦约束**：android-impl 通过 Retrofit + OkHttp + JWT Token 调 backend 仓 REST API；本地缓存用 Room（SQLite）+ WorkManager 同步队列；扫码用 CameraX + ML Kit Barcode。

> 渲染源：PRD `docs/prd/epic-1-基础设施与权限.yaml#E1-S4` + `docs/prd/2-需求.md#FR-1-4`（FR-1-4-1/2/3/4）

### AC-1.4.1 — APP 登录 + Token 持久化 + 指纹解锁（FR-1-4-1）

**GIVEN** 仓管/生管/品管/业务员 已安装 APP（com.btsheng.erp · version 1.0.0），Story 1.1 backend 已就绪 + `POST /api/v1/auth/login` 返回 `access_token` (2h) + `refresh_token` (7d) + `jti` + `roles` + `permissions`
**WHEN** 用户执行下列操作之一：
- (a) 打开 APP → 登录页 → 输入 username/password → 调 `POST /api/v1/auth/login` → 200 → Token 存 EncryptedSharedPreferences（AES-256-GCM）
- (b) 下次打开 APP → 自动读 EncryptedSharedPreferences → 校验 token 未过期 → 直接进首页
- (c) Token 过期（2h 后）→ 静默调 `POST /api/v1/auth/refresh` → 更新 Token → 继续业务
- (d) 登录页 → 勾选"记住密码" → 加密存 username + password（Android Keystore master key）
- (e) 登录页 → 启用指纹解锁（`BiometricPrompt` API 28+）→ 校验通过 → 自动用缓存 Token 进首页

**THEN** 5 个子流程全部通过；登录耗时 P95 < 1.5s（含网络）；Token 加密存储 AES-256-GCM（Android Keystore）；指纹解锁 P95 < 500ms

**AND** Story 1.1 35 测例回归不破（backend `POST /api/v1/auth/login` 已实装 5 类用户 + 5 测例）；`/api/v1/app/login` 端点是 backend Story 1.4 新增的轻量封装（含 `deviceId` + `platform` + `appVersion` + `loginType=PASSWORD/FINGERPRINT`）

### AC-1.4.2 — 消息中心 + 4 类消息 + 离线缓存（FR-1-4-2）

**GIVEN** 用户已登录，Story 1.2 `stream:notify` Redis Stream 推送已就绪（PC 红点 + 邮件 + APP 推送 + 企业微信），Story 1.3 `app.cache-ttl` 系统参数已可读（默认 24h）
**WHEN** 用户执行：
- (a) 打开 APP → 底部 Tab"消息" → 调 `GET /api/v1/app/messages?unreadOnly=true` → 返回 4 类消息（`APPROVAL_NOTIFY` 审批通知 / `OVERDUE_NOTIFY` 逾期提醒 / `EXCEPTION_REPORT` 异常上报 / `SCAN_RECEIPT` 扫码回执）
- (b) APP 收到 `stream:notify` 推送 → 调 `POST /api/v1/app/sync` 增量同步（lastSyncTime 字段）→ Room 本地 SQLite 缓存（按 `app.cache-ttl` 24h 过期）
- (c) 单条消息点击 → 调 `POST /api/v1/app/messages/{id}/read` → 标记已读 + 跳对应业务页（APPROVAL_NOTIFY → 审批详情 / OVERDUE_NOTIFY → 待办列表 / EXCEPTION_REPORT → 异常详情 / SCAN_RECEIPT → 扫码记录）
- (d) 离线时打开消息页 → 读 Room 本地缓存 → 标记"本地缓存（24h 前）"灰显

**THEN** 4 类消息正确分类 + 离线缓存 + 跳转业务页；消息列表分页（每页 20 条）+ 下拉刷新 + 已读未读红点

**AND** Story 1.2 `WorkflowEventPublisher` 4 通道推送 + Story 1.3 系统参数 + Room ORM 缓存 TTL 由 `app.cache-ttl` 控制（可后台 admin 改 TTL）

### AC-1.4.3 — 扫码壳 + 5 类码识别（FR-1-4-3）

**GIVEN** 用户在任一页面点"扫码" → CameraX + ML Kit Barcode 弹出摄像头，5 类码前缀已定义：
- `GD-{yyyyMMdd}-{seq:4}` 工单码（Epic 5 生产）
- `WL-{materialCode}` 物料码（Epic 3 图纸物料）
- `LZ-{yyyyMMdd}-{seq:4}` 流转码（Epic 5 生产 · 工序过站）
- `SB-{deviceCode}` 设备码（Epic 5 生产 · 设备点检）
- `WW-{yyyyMMdd}-{seq:4}` 委外订单码（Epic 6 委外）

**WHEN** 摄像头扫到 5 类码之一：
- (a) 解析前缀 → 调 `GET /api/v1/app/scan/route?code=GD-20260610-0001` → 返回 `{type: WORK_ORDER, id: 1, routeUrl: "workorder/detail/1"}`
- (b) 前端按 `routeUrl` 路由跳转（WorkOrderDetailActivity / MaterialDetailActivity / FlowDetailActivity / DeviceDetailActivity / OutsourceOrderDetailActivity）
- (c) 扫码成功后 → 写 1 条 `SCAN_RECEIPT` 消息到本地 Room（同步到 backend）
- (d) 扫到未知前缀（不是 5 类）→ 弹"未知码类型" + 提供"手动输入"

**THEN** 5 类码 100% 正确识别 + 路由跳转 + 扫码记录留痕；扫码耗时 P95 < 500ms（识别 + 路由 + 跳转）

**AND** Story 1.1 35 + Story 1.2 151 + Story 1.3 19 = 205 测例回归不破（`/api/v1/app/scan/route` 是 backend Story 1.4 新增的路由端点，含 5 类码 prefix 解析）

### AC-1.4.4 — 离线模式 + SQLite 缓存 + 冲突解决（FR-1-4-4）

**GIVEN** 用户在仓库门口/车间等网络不稳定环境，APP 需离线工作
**WHEN** 用户执行：
- (a) 离线时扫码 → 写 Room 本地表 `pending_scan`（含 code + 本地时间戳 + deviceId + 离线标记）
- (b) 离线时打开业务页 → 读 Room 缓存（如 WorkOrder 列表）→ 标记"离线模式 · 数据为 X 分钟前"
- (c) 网络恢复 → WorkManager 周期任务（每 5min）→ 调 `POST /api/v1/app/sync` 上传 `pending_scan` 列表 + 拉服务端最新
- (d) 服务端有更新（服务端时间戳 > 本地时间戳）→ 弹冲突弹窗："服务端有更新，是否覆盖本地？" [覆盖] [合并] [取消]

**THEN** 离线 100% 可用（核心业务如扫码过站、扫码到货不依赖网络）；同步队列 FIFO；冲突弹窗 3 选项 + 用户选择 + 本地标记 `conflict_resolved` + 审计 1 行

**AND** 离线缓存 TTL 由 Story 1.3 `app.cache-ttl` 系统参数控制（默认 24h，可后台改 48h/72h）

## 关键反馈（dev 实施时须注意的）

1. **T0.1 依赖版本**：Kotlin 1.9 + AGP 8.2 + Room 2.6 + CameraX 1.3 + ML Kit Barcode 17.2 + Biometric 1.2 + Hilt 2.50（与 android-impl 已有 `build.gradle.kts` 兼容）
2. **T1.3 加密存储**：必须用 `EncryptedSharedPreferences`（不是普通 SharedPreferences）+ Android Keystore master key + AES-256-GCM（V1.3.7 红线 1）
3. **T2.5 TTL 动态读**：每次 APP 启动调 `GET /api/v1/params/app.cache-ttl` → 缓存到内存（避免每次都查 DB）；TTL 变更后下次启动生效
4. **T3.1 CameraX 权限**：必须 `Manifest.permission.CAMERA` + 运行时权限申请（Android 6.0+）+ 隐私协议
5. **T3.3 prefix 解析**：5 类码 prefix 严格按 PRD 定义（WN/WL/WR/WD/WW），大小写敏感；解析失败 → T3.7 手动输入
6. **T4.3 冲突检测**：服务端 ts 比本地 ts 新（> 5s 容差）才弹窗；避免误报
7. **跨仓集成测试**：android-impl 用 MockWebServer 模拟 backend（不依赖真实 backend 启动），保证 CI 可独立跑
