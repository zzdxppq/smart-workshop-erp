## Epic Context

- **WHY（来自 PRD）**：客户原话"系统要能改金额阈值、字典、编号规则"—— 系统参数是 ERP 的"可配置能力"，业务模块（销售/采购/财务）不写死硬编码值，全部从字典+系统参数读；V1.3.7 红线要求"金额阈值变更走变更日志"。同时 Story 1.2 残余风险 R2/R4/R5 + architect 反馈的"HR Feign 真实集成 + 性能 + 安全 + 审计清理"必须在本 Story 闭环。
- **业务红线**（V1.3.7）：
  ① 数据字典（FR-1-3-1）：物料分类 / 工序类型 / 表处类型 / 班别 / 仓库 / 币种 — 6 类字典，admin 可改，前端用户只读
  ② 系统参数（FR-1-3-2）：单据编号规则（年/月/日/流水 4 段 + 起始值 + 步长）/ 打印模板 / APP 端离线缓存时长（默认 24h）
  ③ 金额阈值全局（FR-1-3-3）：Story 1.1 的 `sys_role.amount_threshold` 升级为 `sys_global_threshold`（按业务类型 + 角色双维度），变更写 `sys_change_log` + 通知工作流引擎热更新
  ④ HR Feign 真实集成：Story 1.2 `HrEmployeeClient` 当前是 mock + Fallback，本 Story 替换为真实 `erp_business.hr` 端 Feign 调用，删除 `HrEmployeeClientFallback` 兜底（仅保留降级为"全员不跳过" + 审计记录）
  ⑤ 性能 + 安全 + 审计清理：k6 P95 实测（路由 50ms / 推送 200ms / 创建 300ms / 待办 1s）+ OWASP ZAP 6 类越权扫描 + `@Scheduled` 实装审计清理（保留期 ≥ 1 年 + JIRA 跟踪）
- **解耦约束**：系统参数服务是 **backend 平台层**（`erp_platform` schema），业务模块通过 `WorkflowConfig` Nacos 模式 + `DictCache` 读字典（V1.3.7 已实装 core/model/DictCache.java）；HR Feign 客户端在 `erp_platform.auth.workflow.feign` 路径，消费 `erp_business` 模块 `POST /api/v1/hr/employees/{id}/status`。

### AC-1.3.1 — 数据字典 CRUD + 6 类字典（FR-1-3-1）

**GIVEN** admin 已登录（具备 `dict:read` / `dict:create` / `dict:update` / `dict:delete` 权限，菜单"系统 → 数据字典"），Story 1.1 已初始化 9 类角色 + Story 1.2 已就绪工作流引擎
