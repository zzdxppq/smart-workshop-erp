## Epic Context

- **WHY（来自 PRD）**：客户原话"业务员不能改单价、经理才能看毛利"—— 权限控住"谁能操作"，审批工作流控住"金额多大需要谁批"。Story 1.1 定义了"金额阈值"和"角色路由"的最小语义（`QuoteApprovalRouter.route()` 返回 `approverUserId` + `currentNode`）；**Story 1.2 升级为完整工作流引擎**（多节点 + 条件分支 + OR 会签 + 超时提醒），后续 Epic 2-销售、5-采购、9-财务 全部依赖此中台。
- **业务红线**（V1.3.7）：① 4 套内置模板（报价/订单/采购/付款）必须可复制可改；② 金额分级口径：`< 5万` 业务员自审 / `5-20万` 部门经理 / `> 20万` 总经理（客户口径，与 Story 1.1 一致）；③ 审批超时 24h 必须推送（PC 红点 + 邮件 + APP 推送 + 企业微信 · 复用 core/redis/RedisStreamTemplate）；④ OR 会签 + 跳过 `hr.status='请假'` 用户（**V1.3.7 P1 修补 · 来自 Story 1.1 architect 评审闭环**）；⑤ 审计 100% 留痕（`@AuditLog` AFTER_COMMIT）。
- **解耦约束**：工作流引擎是 **backend 平台层**（`erp_platform` schema），业务模块（`erp_business` / `erp_production` / `erp_finance`）通过 Feign 客户端 + 事件流（`stream:notify` Redis Stream）消费工作流能力，**严禁** 工作流服务反向依赖业务表。

### AC-1.2.1 — 工作流 CRUD + 4 套内置模板（FR-1-2-1 / FR-1-2-2）

**GIVEN** 管理员已登录（具备 `workflow:create` / `workflow:read` / `workflow:update` 权限），Story 1.1 已初始化 4 类角色（`salesperson` / `dept_manager` / `gm` / `finance`）+ HR 模块已就绪（`hr_employee.status` 字段）
**WHEN** 管理员执行下列操作之一：
- (a) `POST /api/v1/workflows` 创建自定义工作流（`workflow_code=CUSTOM_FLOW` + `nodes_json + conditions_json`）
- (b) `GET /api/v1/workflows` 列出 4 套内置模板（QUOTE_FLOW / ORDER_FLOW / PURCHASE_FLOW / PAYMENT_FLOW）+ 自定义工作流
- (c) `PUT /api/v1/workflows/{id}` 修改某工作流的节点或阈值
- (d) `POST /api/v1/workflows/{id}/test` 试跑工作流（传入测试金额 `amount=80000`），返回匹配路径

**THEN**
1. `POST /api/v1/workflows` 返回 `200 OK`，body `{code:0, msg:"工作流创建成功", data:{id:1, workflowCode:"CUSTOM_FLOW", status:"ACTIVE"}}`；
2. 数据库 `erp_platform.sys_workflow` 新增一行，`workflow_code` 全局唯一（`UNIQUE KEY uniq_workflow_code`），`nodes_json` 是 `List<WorkflowNode>` 序列化结果（V1.3.7 升级：底层新增 `sys_workflow_node` 物理表做细粒度查询，`sys_workflow.nodes_json` 保留为冗余快照，便于 V1.3.4 Stream 重放）；`@AuditLog(module="workflow", action="workflow.create")` 写入 `sys_audit_log`（AFTER_COMMIT 钩子触发）；
3. `GET /api/v1/workflows?pageNum=1&pageSize=20&status=ACTIVE` 返回 4 套内置（`QUOTE_FLOW` / `ORDER_FLOW` / `PURCHASE_FLOW` / `PAYMENT_FLOW`）按 `init.sql:1080-1085` 种子数据 + 自定义工作流，按 `created_at DESC` 排序，**分页总条数**与 SQL `SELECT COUNT(*)` 一致；
4. `PUT /api/v1/workflows/{id}` 修改 `nodes_json` 时校验：① 节点 `node_index` 1..N 严格递增；② 阈值 `threshold` 必须单调非降（否则 `40001 PARAM_INVALID` "阈值必须单调非降"）；③ 节点类型 ∈ {`START` / `APPROVAL` / `CC` / `END`}；④ 首节点必须为 `START`、末节点必须为 `END`；⑤ `WorkflowConfig` 校验不通过时**事务回滚**；
5. `POST /api/v1/workflows/{id}/test` 输入 `{amount: 80000, applicant_role: "salesperson"}`：遍历 `nodes_json` 链，匹配 80000 > 50000（salesperson 阈值）但 ≤ 200000（dept_manager 阈值）→ 路由到 `dept_manager` 节点，返回 `{matchedNodeIndex: 2, matchedRole: "dept_manager", candidates: [10010], trace: [...]}`；试跑结果**不落库**、不产生审批单；
6. 4 套内置模板**不可物理删除**（`DELETE /api/v1/workflows/{id}` 对 `workflow_code IN (QUOTE_FLOW, ORDER_FLOW, PURCHASE_FLOW, PAYMENT_FLOW)` 返回 `40904 BUILTIN_WORKFLOW`），但**可复制**：内置模板"复制改"操作产生新 `workflow_code` + 继承全部 `nodes_json` + `conditions_json` + `version=v1`（V1.3.7 留版本号字段给后续灰度）。

**Business Rules**
- BR-1 `workflow_code` 全局唯一（DB 层 `UNIQUE KEY uniq_workflow_code` + 应用层 `lower(workflow_code)` 比较）
- BR-2 节点阈值单调非降：`< 5万` → `[5万, 20万]` → `> 20万`（NULL 表示无限额）；违反时 `40001 PARAM_INVALID` 拒绝
- BR-3 4 套内置模板（`QUOTE_FLOW` / `ORDER_FLOW` / `PURCHASE_FLOW` / `PAYMENT_FLOW`）不可物理删除（软删 status=DELETED 也禁止），仅可"复制改"产生新模板
- BR-4 节点类型枚举：`START`（node_index=1）/ `APPROVAL`（含 `role_code` + `threshold` + `or_sign_required`）/ `CC`（抄送，可选）/ `END`（node_index=N）
- BR-5 `conditions_json` 字段：V1.3.7 必传 `amount_field`（业务表单的金额字段路径，如 `total_amount`），可选 `extra_check`（如 `credit_limit`）
- BR-6 `nodes_json` AES-256-GCM 加密存储（V1.3.7 红线 · 复用 `core/crypto/AesGcmTypeHandler`），响应体返回明文
- BR-7 角色编码必须在 `sys_role` 存在（`role_code IN (salesperson, dept_manager, gm, finance, ...)`），不存在时 `40401 ROLE_NOT_FOUND`
- BR-8 `or_sign_required` 字段（**V1.3.7 P1 修补**）：多审批人时是否全员会签，默认 `false`（按 Story 1.1 简化版 `user_id ASC LIMIT 1`）；开启后升级为 AC-1.2.3 OR 会签

**Data Validation**
- `workflow_code`：必填，`^[A-Z_]{2,50}$`（全大写 + 下划线，与 init.sql 种子保持一致）
- `nodes_json`：必填，JSON 数组 1..20 个节点
- `conditions_json`：可选 JSON 对象，`amount_field` 必填
- `status`：枚举 `ACTIVE` | `INACTIVE` | `DELETED`（默认 `ACTIVE`）
- `version`：正整数，默认 `1`（V1.3.7 留 hook）

**Error Handling**
| 错误码 | 触发条件 | 错误消息 | HTTP |
|--------|----------|----------|------|
| 40001 | `nodes_json` 阈值不单调 | "节点阈值必须单调非降" | 400 |
| 40002 | 节点类型非法 | "节点类型必须为 START/APPROVAL/CC/END" | 400 |
| 40003 | 首节点非 START | "首节点必须为 START" | 400 |
| 40004 | 末节点非 END | "末节点必须为 END" | 400 |
| 40005 | 节点数 < 2 或 > 20 | "节点数必须在 2-20 之间" | 400 |
| 40401 | 工作流不存在 | "工作流 {id} 不存在" | 404 |
| 40401 | 角色不存在 | "角色 {role_code} 不存在" | 404 |
| 40904 | 删除内置模板 | "内置工作流不可删除，请使用复制改" | 409 |
| 40905 | `workflow_code` 重复 | "工作流编码 {code} 已存在" | 409 |
| 40906 | 状态机不匹配（修改被锁工作流） | "工作流正在被使用，禁止修改" | 409 |
| 50001 | DB 唯一索引冲突（兜底） | "工作流编码已存在" | 500 |

**Examples**
```json
// Valid - 创建自定义工作流
POST /api/v1/workflows
{
  "workflowCode": "CUSTOM_FLOW",
  "nodesJson": "[{\"node\":1,\"type\":\"START\"},{\"node\":2,\"type\":\"APPROVAL\",\"role\":\"salesperson\",\"threshold\":50000},{\"node\":3,\"type\":\"APPROVAL\",\"role\":\"dept_manager\",\"threshold\":200000},{\"node\":4,\"type\":\"END\"}]",
  "conditionsJson": "{\"amount_field\":\"total_amount\"}"
}
// → 200 {code:0, data:{id:1, workflowCode:"CUSTOM_FLOW"}}

// Valid - 试跑工作流
POST /api/v1/workflows/1/test
{ "amount": 80000, "applicantRole": "salesperson" }
// → 200 {code:0, data:{matchedNodeIndex:3, matchedRole:"dept_manager", candidates:[10010]}}

// Invalid - 阈值不单调
{ "nodesJson": "[{\"node\":1,\"type\":\"APPROVAL\",\"threshold\":200000},{\"node\":2,\"type\":\"APPROVAL\",\"threshold\":50000}]" }
// → 40001 "节点阈值必须单调非降"

// Invalid - 删除内置模板
DELETE /api/v1/workflows/1   // workflow_code = QUOTE_FLOW
// → 40904 "内置工作流不可删除，请使用复制改"
```

### AC-1.2.2 — 金额阈值实时路由 + 审批单创建（FR-1-2-1 核心 + Story 1.1 升级）

**GIVEN** 业务员「张三」（`user_id=10086`）已登录（仅 `salesperson` 角色），Story 1.1 阈值已就绪：
- `salesperson.amount_threshold = 50000`
- `dept_manager.amount_threshold = 200000`
- `gm.amount_threshold = NULL`（无限额）
- `dept_manager` 角色下存在 2 个 ACTIVE 用户：`user_id=10010`（张三上司「李四」）、`user_id=10011`（市场部「王五」）

业务侧 `erp_business` 报价模块调用 `POST /api/v1/business/quotes` 提交报价单（`BJ202606100001`，`total_amount=60000.00`，`customer_type=VIP`）

**WHEN** 系统创建报价单后调用 `ApprovalService.createApproval(bizType=QUOTE, bizId=BJ202606100001, amount=60000, applicantUserId=10086)`

**THEN**
1. **路由计算**（**升级** Story 1.1 `QuoteApprovalRouter`）：调用 `QuoteApprovalRouter.route(amount=60000, applicantUserId=10086, workflowCode=QUOTE_FLOW)` → 60000 > salesperson 阈值 50000 → 跳过业务员自审 → 命中 `dept_manager` 节点（50000 < 60000 ≤ 200000）→ 取 `dept_manager` 角色下所有 ACTIVE 用户 `candidates=[10010, 10011]`（**V1.3.7 P1 修补**：不再 `LIMIT 1`，返回完整 candidates 列表）；
2. **OR 会签判断**（**V1.3.7 P1 修补**）：读取 `sys_workflow_node.or_sign_required` 字段（来自 `nodes_json` 反序列化或 `sys_workflow_node` 物理表）→ 本例 `QUOTE_FLOW` 第 2 节点 `or_sign_required=false`（默认）→ 采用"**任一会签即通过**"语义：10010 任一人审批通过 → 流程推进到下一节点（gm）；
3. **写入审批单**：`erp_platform.sys_approval_record` 新增 1 条记录（`biz_type=QUOTE`、`biz_id=BJ202606100001`、`workflow_code=QUOTE_FLOW`、`current_node_index=2`、`current_approver_user_id=10010`（取 `candidates[0]`，**首次分配确定性**）、`status=PENDING`、`created_at=NOW`）；同时为 10011 写一条 `status=WAITING` 的占位记录（V1.3.7 OR 会签：允许任一人审批，但所有人都有占位便于审计追溯）；
4. **状态机守卫**：`sys_workflow.status` 必须为 `ACTIVE`（V1.3.7 字段）；若 status=`INACTIVE` 则 `ApprovalService.createApproval` 抛 `BizException(40906 WORKFLOW_INACTIVE)`，**事务回滚**不创建审批单；
5. **超时基准时间**：`created_at + 24h` 写入 `sys_approval_record.timeout_at`（V1.3.7 字段 · FR-1-2-3 配合）；若超过 24h 仍未处理 → AC-1.2.4 推送 + 标记 `is_overdue=true`；
6. **审计 100%**：`@AuditLog(module="approval", action="approval.create")` AFTER_COMMIT 写入 `sys_audit_log`，`after_json` 含 `biz_id / current_approver_user_id / or_sign_required / candidates_size`；
7. **通知实时推送**（复用 `core/redis/RedisStreamTemplate`）：向 `stream:notify` 发送 `XADD stream:notify * bizType APPROVAL bizId BJ202606100001 targetUserId 10010,10011 title "待审批：报价单 BJ202606100001 金额 ¥60000" ts 1718016000`；下游通知消费者（`notification-consumer` 模块，留 Story 1.4 APP 端实施）自动消费并推送 APP/邮件/企业微信。

**Business Rules**
- BR-9 路由算法（**升级** Story 1.1）：遍历 `sys_workflow_node` 链（按 `node_index ASC`），对 `amount` 与 `threshold` 做 `compareTo`（`BigDecimal` 严格大于才进入下一节点），NULL 阈值为兜底（无限额）
- BR-10 首次分配确定性：多审批人时 `candidates` 按 `user_id ASC` 排序，取 `[0]` 作为 `current_approver_user_id`（保证张三每次都看到同一个审批人，避免抖动）
- BR-11 OR 会签语义（**V1.3.7 P1 修补**）：`or_sign_required=true` → 任一候选人 `approve` 即可推进；`or_sign_required=false` → 沿用 Story 1.1 简化版（按 `user_id ASC` 第 1 个有效用户）
- BR-12 状态机：`PENDING` → `APPROVED` / `REJECTED` / `SKIPPED`（V1.3.7 跳过 `hr.status='请假'` 用户时设为 SKIPPED）；禁止 `PENDING → PENDING` 自循环（状态机不匹配 → 40904）
- BR-13 审批超时：`timeout_at = created_at + 24h`；超过 24h 未处理 → AC-1.2.4 推送 + `is_overdue=true`（不自动驳回，留人工干预）
- BR-14 金额比较 `BigDecimal.compareTo`：严禁 `equals`（避免精度问题）；统一 `(a.compareTo(b) > 0)` 表示 `a > b`
- BR-15 路由结果不缓存：每次实时读 `sys_role.amount_threshold` 和 `sys_workflow_node`（角色调整后立即生效）
- BR-16 `or_sign_required` 字段缺失时按 `false` 处理（向后兼容 Story 1.1 路由结果）

**Data Validation**
- `biz_type`：必填，枚举 `QUOTE` | `ORDER` | `PURCHASE` | `PAYMENT`
- `biz_id`：必填，字符串 ≤ 50 字符（业务单号）
- `amount`：必填，`DECIMAL(15,2)`，`> 0`
- `applicant_user_id`：必填，`BIGINT`，必须在 `sys_user` 存在
- `workflow_code`：必填，必须在 `sys_workflow` 存在且 `status=ACTIVE`
- `comment`：可选，`^.{0,500}$`
- `reason`：驳回时必填，`^.{1,500}$`

**Error Handling**
| 错误码 | 触发条件 | HTTP |
|--------|----------|------|
| 40006 | `biz_type` 不在枚举 | 400 |
| 40007 | `amount` 格式错误（非数字/负数/0） | 400 |
| 40008 | `workflow_code` 不存在或非 ACTIVE | 400 |
| 40401 | 审批人不存在 | 404 |
| 40402 | 角色下找不到任何 ACTIVE 候选人 | 404 |
| 40303 | 申请人无 workflow 关联的任一角色 | 403 |
| 40904 | 状态机不匹配（如 PENDING→PENDING） | 409 |
| 40906 | 工作流 INACTIVE | 409 |
| 50002 | Redis Stream 推送失败（兜底重试） | 500 |

**Examples**

### AC-1.2.3 — OR 会签 + 跳过请假用户（V1.3.7 P1 修补 · 来自 Story 1.1 architect 评审）

**GIVEN** AC-1.2.2 Case 2 已创建审批单 `approval_id=2`（`candidates=[10010, 10011]`，`or_sign_required=false`），其中：
- `user_id=10010`（李四 · 部门经理）当日在岗
- `user_id=10011`（王五 · 部门经理）当日 **请假**（`hr_employee.status='请假'`，请假单 HR-2026-06-10-001 已审批）

**WHEN** 流程推进触发 `SkipOnLeaveRule.evaluate(approvalId=2, candidates=[10010, 10011])`（**V1.3.7 P1 修补** · 复用 Story 1.1 `QuoteApprovalRouter.candidates: List<Long>` 字段）

**THEN**

5. **审计 100%**：`@AuditLog` AFTER_COMMIT 覆盖 `WorkflowService` CRUD + `ApprovalService` 创建/approve/reject/skip/urge
6. **Redis Stream 复用**：复用 `core/redis/RedisStreamTemplate`（V1.3.4 委外 7 状态机消息发布基础设施）—— 通道 `stream:notify` + `stream:app-push`

9. **状态机不匹配 → 40904**：`PENDING → APPROVED/REJECTED/SKIPPED`；禁止 `PENDING → PENDING` 自循环；同审批人重复 approve 抛 40904
10. **金额超限 → 40303**：审批人不属于 `candidates` 列表 + 申请人无 workflow 关联的任一角色 → 40303
11. **工作流不存在 → 40401**：40401（工作流）/ 40401（角色）/ 40401（审批人）/ 40402（角色下找不到任何 ACTIVE 候选人）
