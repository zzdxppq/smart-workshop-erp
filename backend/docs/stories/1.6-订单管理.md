## Epic Context

- **WHY（来自 PRD）**：订单是销售流程的"金额固化点"——报价是"虚的"（可改），订单是"实的"（要生产、要回款）。业务员原话"客户说改数量我能改吗？"—— 订单既要支持数量调整（来自报价的 `quantityAdjustment`），也要支持信用额度校验（业务员超信用额度的订单要总经理审批），还要支持一键转生产（Epic 5）+ 一键转委外（Epic 6）。
- **业务红线**（V1.3.7）：
  ① 订单号 `XS+年月日+4位流水`（如 `XS20260612-0001`），由 Story 1.5 `DocNoGenerator` 复用（`biz.doc-no.order` 模板）
  ② 状态机 7 节点：`DRAFT → CONFIRMED → PRODUCING → PARTIAL_SHIPPED → SHIPPED → SETTLED → CLOSED`（V1.3.7 §附录-b 状态机速查表）
  ③ 信用额度校验：调 Story 1.3 `sys_dict?type=CREDIT_LIMIT` 查客户信用额度 → 超出抛 40909 `CREDIT_LIMIT_EXCEEDED`（V1.3.7 P2 修补 3 的 hook）
  ④ 一键转生产（Epic 5）+ 一键转委外（Epic 6），订单状态自动推进
  ⑤ 订单导出 PDF（带公司 logo + 订单明细 + 审批签字栏）+ Excel
- **解耦约束**：订单服务是 `erp_business` schema 业务模块（与 `erp_platform` 工作流中台解耦），通过 **Feign 客户端** + **Redis Stream 事件**消费工作流能力

## Acceptance Criteria

> 渲染源：PRD `docs/prd/epic-2-客户与销售.yaml#2.3` + `docs/prd/2-需求.md#FR-2-3`（FR-2-3-1/2/3/4/5/6）

### AC-2.3.1 — 订单 CRUD + 字段必填校验（FR-2-3-1）

**GIVEN** 业务员已登录（具备 `order:create/order:read/order:update` 权限），Story 1.5 `OrderConversionService` 已实装
**WHEN** 业务员执行下列操作之一：
- (a) `POST /api/v1/orders` 创建订单（手动创建或经 `/api/v1/quotes/{id}/convert-to-order` 转单，请求体含 `customerId` + `deliveryDate` + `currency=CNY` + `items[]` 图号/材质/尺寸/数量/单价/金额/是否FA/是否新件 + `comment` + `quantityAdjustment` 可选）
- (b) `GET /api/v1/orders/{id}` 查询订单详情（含状态机 + 历史修改记录 + 转生产/委外链路）
- (c) `PUT /api/v1/orders/{id}` 修改订单（仅 `DRAFT` 状态可改；`CONFIRMED` 及之后状态抛 40903 `ORDER_NOT_EDITABLE`）
- (d) `GET /api/v1/orders?page=0&size=20&status=&customerId=&owner=&dateFrom=&dateTo=` 列表查询（按权限过滤：业务员见自己 / 经理见本部门 / 总经理见全部）

**THEN** 8 端点全部通过；订单号 `XS{yyyyMMdd}{seq:4}` 实时生成（继承 1.5 `DocNoGenerator`）；字段校验（`customerId` 必填 + `items` ≥ 1 + 数量 > 0 + 单价 ≥ 0 + 交期 ≥ 今天 + `isFa`/`isNew` 是 boolean）；CRUD 100% 留痕（`@AuditLog` AFTER_COMMIT）

**AND** 信用额度校验：调 Story 1.3 `sys_dict?type=CREDIT_LIMIT` 查客户 `creditLimit`，`sum(items.amount) > creditLimit` → 抛 40909 `CREDIT_LIMIT_EXCEEDED`（V1.3.7 P2 修补 3 的 hook）；金额自动计算：`amount = sum(item.quantity * item.unitPrice)`，存到 `order.total_amount`（DECIMAL(15,2)）

### AC-2.3.2 — 状态机推进 + 工作流集成（FR-2-3-2）

**GIVEN** 订单已创建（`DRAFT` 状态），Story 1.2 4 工作流模板 + 1.5 `OrderApprovalRouter` 已就绪
**WHEN** 业务员或系统执行：
- (a) `POST /api/v1/orders/{id}/confirm` 确认订单（status `DRAFT → CONFIRMED`，触发 Story 1.2 4 通道推送 `stream:notify` + `OrSignRequired=false`）
- (b) 客户为黑名单 → 直接驳回 40902 `CUSTOMER_BLACKLIST`，不进入确认流（复用 1.5 黑名单校验）
- (c) 金额 `< 5万` → 业务员自审（`salesperson` 角色）
- (d) 金额 `5-20万` → 部门经理审批（`dept_manager` 角色，可能 OR 会签多人）
- (e) 金额 `> 20万` → 总经理审批（`gm` 角色）+ 财务总监双签（V1.3.7 `PAYMENT_FLOW` 模板）
- (f) `POST /api/v1/orders/{id}/approve` 审批通过（status `CONFIRMED → PRODUCING`，触发 `stream:notify` 推送给业务员 + Epic 5 生产工单生成）
- (g) `POST /api/v1/orders/{id}/reject` 驳回（请求体含 `reason`，status `CONFIRMED → DRAFT`（驳回回退到 DRAFT，可重新编辑），通知业务员 + `@AuditLog` 写 `order.reject_reason`）

**THEN** 4 阈值路由 100% 准确（与 Story 1.5 `QuoteApprovalRouter` 兼容）；`skipOnLeaveRule` 跳过请假候选人；OR 会签候选 `candidates: List<Long>` 完整列表返回前端；`@AuditLog` 100% 留痕

**AND** 状态机守卫：7 状态严格推进（V1.3.7 §附录-b）：
- `DRAFT → CONFIRMED`（可驳回回退到 DRAFT）
- `CONFIRMED → PRODUCING`（不可逆）
- `PRODUCING → PARTIAL_SHIPPED`（部分发货，quantity_shipped < quantity）
- `PRODUCING → SHIPPED`（全部发货，quantity_shipped == quantity）
- `PARTIAL_SHIPPED → SHIPPED`（剩余部分全部发货）
- `SHIPPED → SETTLED`（财务收款完成，引用 Epic 9 `crm_receipt`）
- `SETTLED → CLOSED`（订单最终关闭，不可再操作）

### AC-2.3.3 — 信用额度校验（V1.3.7 P2 修补 3 的 hook · FR-2-3-3）

**GIVEN** 订单已 `DRAFT`，客户档案（Story 2.1）已含 `creditLimit` 字段，Story 1.3 `sys_dict` 已就绪
**WHEN** 业务员或系统执行 `POST /api/v1/orders` 或 `POST /api/v1/quotes/{id}/convert-to-order`：
- (a) 订单金额 `<= creditLimit` → 校验通过，进入状态机
- (b) 订单金额 `> creditLimit` → 抛 40909 `CREDIT_LIMIT_EXCEEDED`，状态保持 `DRAFT`
- (c) 客户在黑名单（V1.3.7 红线 1）→ 直接抛 40902 `CUSTOMER_BLACKLIST`（优先于信用额度校验）
- (d) 客户为 `creditLimit = -1`（无限制）→ 跳过校验（V1.3.7 字典特殊值）

**THEN** 信用额度校验在订单创建/转单/编辑时强制执行；`creditLimit` 字段由 Story 2.1 客户档案维护；`@AuditLog` 写 `order.credit_check` 留痕

**AND** 与 Story 1.5 报价信用额度 hook 协同：报价 `APPROVED` 转订单时如 `order.totalAmount > customer.creditLimit` → 抛 40909，订单保持 `DRAFT`，由业务员决定是否调小数量或申请信用额度调整（提交 admin 审批，V1.3.7 §附录-a 金额阈值速查表）

### AC-2.3.4 — 转生产 + 转委外 + PDF/Excel 导出（FR-2-3-4/5/6）

**GIVEN** 订单已 `CONFIRMED`（或 `PRODUCING`），Story 1.5 `OrderConversionService` 已实装
**WHEN** 业务员执行：
- (a) `POST /api/v1/orders/{id}/convert-to-production` 转生产工单（status `CONFIRMED → PRODUCING`，触发 Epic 5 `ProductionOrderService.createProductionOrder`，生成工单号 `GD{yyyyMMdd}{seq:4}`）
- (b) `POST /api/v1/orders/{id}/convert-to-outsource` 转委外单（status `CONFIRMED → PARTIAL_SHIPPED`（部分委外）或 `PRODUCING`（部分生产 + 部分委外），触发 Epic 6 `OutsourceOrderService.createOutsourceOrder`，生成委外单号 `WW{yyyyMMdd}{seq:4}`）
- (c) `POST /api/v1/orders/{id}/ship` 发货（`quantityShipped` 累计，status `PRODUCING → PARTIAL_SHIPPED` 或 `PRODUCING → SHIPPED`，触发 APP 推送 `stream:app-push` 给客户联系人）
- (d) `GET /api/v1/orders/export/{id}?format=pdf` 导出 PDF（带公司 logo + 订单明细 + 审批签字栏 + 审批时间戳）
- (e) `GET /api/v1/orders/export/{id}?format=excel` 导出 Excel（多 Sheet：订单基本信息 + 订单明细 + 审批记录 + 发货记录）
- (f) PDF 导出后 1h 内可重复下载（`app.cache-ttl` 控制 PDF 文件缓存 + `crm_audit_log` 写 `pdf.download` 留痕）

**THEN** 转生产/转委外 100% 字段自动带出（图纸/明细/数量/单价）；发货数量累计校验（`sum(quantityShipped) <= item.quantity`）；PDF 用 `OpenPDF 1.3.34` 生成（继承 1.5 选型）；Excel 用 `Apache POI 5.2.5` 生成；`@AuditLog` 写 `order.convert_to_production/outsource/ship/pdf_download`

**AND** 5 条 UI 红线（V1.3.7 §8 · 继承 1.5 4 条 + 新增 1 条）：① 不显示金额（业务自审 < 5万） ② 二次密码确认（> 20万） ③ 黑名单高亮 ④ **转生产预览**（V1.3.7 新增） ⑤ **转委外预览**（V1.3.7 新增）
