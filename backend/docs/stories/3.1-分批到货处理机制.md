## 1. V1.3.7 → V1.3.8 升级动因（PM 范蠡反馈 #2）

PM 反馈：CNC 加工厂来料**不会一次性到齐**，可能分 3-5 批。V1.3.7 按 PO 粒度处理导致：
- 仓管必须等全部到齐才能扫码入库
- 第一批到货无法触发质检
- PO 状态机无法表达"部分到货"

**V1.3.8 方案**：批次号作为最小单元，PO 状态扩展为 3 态（全部到货 / 部分到货 / 待发货），扫码按物料触发质检。

### 1.1 PM 建议 vs V1.3.7 现状

| PM 建议 | V1.3.7 现状 | V1.3.8 缺口 |
|---------|-------------|-------------|
| 扫 PO 二维码 CG- | 1.50 已支持 | 无 |
| 按物料生成来料检单 LJ- | 1.35 按 PO | **重构为按物料** |
| PO 状态 3 态 | 1.34 PENDING/ALERT/ARRIVED | 扩展 `ALL_ARRIVED` / `PARTIAL_ARRIVED` / `PENDING_SHIP` |
| 批次号 YYYYMMDD-流水 | 未实装 | **新增 crm_batch + DocNoGenerator 扩展** |
| 物料码 = 批次号 + WL-XXXX | 1.11 WL-XXXX 单一 | 由 Story 3.2 扩展为复合 |

### 1.2 PO 状态机扩展

```
V1.3.7:
  PENDING → ALERT → ARRIVED (单次到齐)

V1.3.8:
  PENDING_SHIP → PARTIAL_ARRIVED → ALL_ARRIVED
       ↑                ↓
       └── 重新发货 ────┘  (部分到货后供应商补发)
```

## 2. Acceptance Criteria

### AC-3.1.1 批次号生成 + 物料粒度到货

**Given** 仓管扫码 PO 二维码 CG-20260613-0001，系统识别该 PO 含 3 个物料（M1×100, M2×200, M3×50）
**When** 调用 `POST /api/v1/incoming/batch-create` 传入 `[{material_id: M1, quantity: 60, arrived_at: ...}, ...]`
**Then** 系统按物料生成 3 个批次号 `BATCH-20260613-0001/-0002/-0003`
**And** 每个批次写入新表 `crm_batch`
**And** PO 状态从 `PENDING_SHIP` → `PARTIAL_ARRIVED`
**And** 物料 M1 触发来料检单 `LJ-20260613-0001`（按物料，非按 PO）
**And** 双写：旧表 `crm_incoming_record` 同步写入 1 条记录（PO 粒度汇总）

### AC-3.1.2 PO 状态机扩展

**Given** PO `PO-001` 状态为 `PARTIAL_ARRIVED`（M1×60/100 已到）
**When** 仓管继续扫码到货 M1×40 + M2×200 + M3×50
**Then** 系统识别 M1 累计 100/100 → 该物料标记 `ALL_ARRIVED`
**And** M2/M3 累计达 100% → 全部物料完成
**And** PO 状态 → `ALL_ARRIVED`
**And** 触发入库通知 stream:notify 给仓管 + 总经理

### AC-3.1.3 物料粒度扫码 + 灰度

**Given** V1.3.8 代码就位，影子表 `crm_batch` 已建
**When** Sprint 7 IMPL 阶段起，所有 1.34/1.35/1.50/1.51 写入操作
**Then** **双写**：新表 + 老表同时写入
**And** 双读对比：每 1h cron job 对比两表一致性（按 batch_no + material_id 聚合）
**And** 不一致率 > 0.1% 触发告警 stream:notify
**And** 灰度比例：Sprint 7 内 0%（仅双写对比）→ 7/30 FAT 通过后 50% → 8 月底 100%

## 3. 数据库迁移（V49）

### V49__batch.sql

```sql
CREATE TABLE crm_batch (
  id BIGINT NOT NULL AUTO_INCREMENT,
  batch_no VARCHAR(30) NOT NULL COMMENT 'BATCH-YYYYMMDD-流水',
  material_id BIGINT NOT NULL,
  po_id BIGINT NOT NULL,
  po_item_id BIGINT NOT NULL,
  quantity INT NOT NULL,
  arrived_at DATETIME NOT NULL,
  quality_status ENUM('PENDING','PASSED','REJECTED') DEFAULT 'PENDING',
  created_by BIGINT NOT NULL,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (id),
  UNIQUE KEY uniq_batch_no (batch_no),
  KEY idx_material (material_id),
  KEY idx_po (po_id),
  KEY idx_arrived (arrived_at)
) ENGINE=InnoDB COMMENT='物料批次表 V1.3.8';

-- 影子表（与 crm_incoming_record 对比）
CREATE TABLE crm_batch_shadow LIKE crm_batch;

-- PO 状态机扩展
ALTER TABLE crm_purchase_order
  MODIFY COLUMN status ENUM(
    'PENDING_SHIP','PARTIAL_ARRIVED','ALL_ARRIVED',
    'CANCELLED'
  ) NOT NULL DEFAULT 'PENDING_SHIP';
```

## 5. 灰度方案（**代码就位即双写**）

### 5.1 阶段 1：影子表 + 双写（Sprint 7 IMPL 起 · 4 周）

```
写入: [crm_incoming_record (old)] + [crm_batch (new)] + [crm_batch_shadow (compare)]
读: 全部读老表（保证线上 0 影响）
对比: 每 1h cron 对比 crm_batch vs crm_batch_shadow
```

### 5.2 阶段 2：切流 50%（V1.3.8 FAT 通过后 · 7/30 起 · 1 周）

```
写入: 双写不变
读: 50% 流量走新表（按 user_id % 2）
对比: 不一致率 < 0.1% 继续；> 0.1% 回滚 0%
```

### 5.3 阶段 3：全量切流（8 月中 · 1 周）

```
读: 100% 流量走新表
```

### 5.4 阶段 4：清理（8 月底 · 1 周）

```
删除 crm_incoming_record (old) + 老代码路径
保留 crm_batch_shadow 30 天后删除
```
