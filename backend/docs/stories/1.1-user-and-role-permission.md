### AC-1.1.1 — 管理员创建用户并分配角色（FR-1-1-1）

**GIVEN** 管理员已登录系统（具备 `user:create` 权限），并已在角色管理维护好"业务员"角色（`role_code = salesperson`）
**WHEN** 管理员创建新用户「张三」，参数 `username=zhangsan`、`password=Pass1234`、`real_name=张三`、`phone=13800000000`、`role_codes=["salesperson"]`
**THEN**
1. 系统返回 `200 OK`，body `{code:0, msg:"用户创建成功", data:{id:10086, username:"zhangsan"}}`；
2. 数据库 `erp_platform.sys_user` 新增一行，`password_hash` 为 BCrypt(cost=12) 密文（`$2a$12$...`，长度 60 字节），**严禁明文落库**；
3. `sys_user_role` 新增一行 `(user_id=10086, role_id=<salesperson.id>)`；
4. 张三用 `zhangsan` / `Pass1234` 立即可登录，登录后 `/api/v1/auth/me` 返回的 `roles = ["业务员"]`、`permissions` 仅含 `crm:view`、`quote:view`、`order:view`；
5. 张三调用 `GET /api/v1/purchase/orders/{id}` 试图查看采购单，**看不到采购单价、毛利率、合同金额**（接口返回 `40301 无权限`，字段在 mapper 层级就不查询，不只是前端隐藏）。

**Business Rules**
- BR-1 用户名全局唯一（不区分大小写，DB 层 `UNIQUE KEY uniq_username` + 应用层 `lower(username)` 比较）
- BR-2 密码 ≥ 8 位，必须同时含大小写字母 + 数字（正则 `^(?=.*[a-z])(?=.*[A-Z])(?=.*\d).{8,}$`）
- BR-3 密码 BCrypt(cost=12) 加密；`password_hash` 字段严禁写日志、严禁回显到前端
- BR-4 用户被禁用（`status=DISABLED`）后，JWT 立即失效：Redis 黑名单 key `auth:blacklist:{userId}` 写入当前 jti，TTL = access_token 剩余有效期；用户后续请求 5 分钟内必被踢出（`40302 TOKEN_BLACKLISTED`）
- BR-5 登录失败 5 次锁 30 分钟：Redis key `auth:login:fail:{username}` 计数，TTL 30 min 滚动；超限后 `POST /auth/login` 返回 `40103 ACCOUNT_LOCKED`
- BR-6 部门/职位与角色解耦（FR-1-1-4）：同一 `user_id` 可对应多个 `role_id`，按模块取并集；`dept_id` / `position_id` 不影响权限计算

**Data Validation**
- `username`：必填，`^[A-Za-z0-9_]{3,20}$`
- `real_name`：必填，`^.{1,50}$`
- `phone`：可选，11 位数字（`^1[3-9]\d{9}$`）；**存库前用 `@EncryptedField("phone")` + `AesGcmTypeHandler` AES-256-GCM 加密**，DB 存 `iv(12B) + cipher` 二进制
- `email`：可选，符合 RFC 5322
- `role_codes`：必填数组，至少 1 个；为空时返回 `40001 PARAM_INVALID("请至少分配一个角色")`
- `dept_id`：可选，引用 `sys_dept.id`；不存在时返回 `40401 DEPT_NOT_FOUND`

**Error Handling**
| 错误码 | 触发条件 | 错误消息 | HTTP |
|--------|----------|----------|------|
| 40001 | 用户名长度/字符非法 | "用户名仅支持 3-20 位字母数字下划线" | 400 |
| 40002 | 密码强度不足 | "密码必须 ≥ 8 位含大小写字母+数字" | 400 |
| 40001 | 角色 codes 数组为空 | "请至少分配一个角色" | 400 |
| 40003 | 手机号格式错误 | "手机号必须为 11 位数字" | 400 |
| 40901 | 用户名重复 | "用户名已存在" | 409 |
| 40401 | 角色编码不存在 | "角色 salesperson 不存在" | 404 |
| 50001 | DB 唯一索引冲突 | "用户名已存在"（兜底） | 500 |

**Examples**
```json
// Valid
{ "username": "zhangsan", "password": "Pass1234", "real_name": "张三",
  "phone": "13800000000", "email": "zhangsan@byts.com",
  "dept_id": 1, "role_codes": ["salesperson"] }

// Invalid 1 - 用户名过短
{ "username": "zs", "password": "Pass1234", ... }
// → 40001 "用户名仅支持 3-20 位字母数字下划线"

// Invalid 2 - 密码仅数字
{ "username": "lisi", "password": "12345678", ... }
// → 40002 "密码必须 ≥ 8 位含大小写字母+数字"

// Invalid 3 - 重复用户名
{ "username": "zhangsan", ... }  // zhangsan 已存在
// → 40901 "用户名已存在"
```

### AC-1.1.2 — 角色权限管理（菜单/数据/操作）（FR-1-1-2）

**GIVEN** 管理员已登录并进入「角色管理」页
**WHEN** 管理员编辑「业务员」角色（`role_id=10`），将"订单模块 → 客户列表"项的 `view_all` 操作权限从"开"改为"关"，保存
**THEN**
1. 接口 `PUT /api/v1/roles/10` 返回 `200 OK`；
2. DB `sys_role_permission` 中 `(role_id=10, menu_id="order.customer", action="view_all")` 行被物理删除；
3. 变更写入 `platform_audit_log`（`@AuditLog` 注解自动记录前值 `["view_all:on"]`、后值 `["view_all:off"]`、操作人 admin、操作时间、客户端 IP）；
4. 所有"业务员"角色用户（实时）登录后，调用 `GET /api/v1/auth/me` 返回的 `permissions` 数组不再包含 `order.customer:view_all`；
5. 业务员「张三」访问"客户列表"页：仅能看到 `creator_id = 张三.id` 的客户（"我的客户"），看不到他人创建的客户（"全部客户"菜单项自动隐藏）；后端 `GET /api/v1/customers?scope=all` 强制校验 `order.customer:view_all` 权限，无权则返回 `40301`；
6. 角色删除时若有 `sys_user_role` 引用：`DELETE /api/v1/roles/10` 返回 `40902 ROLE_IN_USE` "角色已被 N 个用户引用，禁止删除"。

**Business Rules**
- BR-7 菜单权限：树形勾选（父节点半选=部分子节点被勾选，全选/全不选/反选快捷按钮）
- BR-8 数据权限（4 级，写在 `sys_role.data_scope`）：`SELF`（仅本人）/ `DEPT`（本部门）/ `ALL`（全部）/ `CUSTOM`（自定义部门列表，写在 `sys_role_permission` 扩展表，V1.3.7 暂以 ALL 兜底）
- BR-9 操作权限：6 维独立勾选 —— 增（`add`）/ 改（`edit`）/ 删（`delete`）/ 审（`audit`）/ 导（`export`）/ 打（`print`）
- BR-10 角色编码 `role_code` 全局唯一（`UNIQUE KEY uniq_role_code`）；名称 `role_name` 1-20 字符
- BR-11 角色删除前必须校验 `sys_user_role` 引用计数 = 0，否则拒绝
- BR-12 内置角色（`role_code IN (admin, salesperson, dept_manager, gm, finance, purchasing, warehouse, quality, production_planner)`）不可禁用/删除；业务员、经理、总经理三类**必须存在**

**Data Validation**
- `role_code`：必填，`^[a-z_]{2,50}$`，全小写
- `role_name`：必填，`^.{1,20}$`
- `data_scope`：枚举 `SELF | DEPT | ALL | CUSTOM`
- `amount_threshold`：`DECIMAL(15,2)`，可空，0 < x < 999,999,999,999.99

**Error Handling**
| 错误码 | 触发条件 | HTTP |
|--------|----------|------|
| 40902 | 删除已被引用的角色 | 409 |
| 40903 | 修改内置角色 code/name | 409 |
| 40004 | 权限 menu_id 不存在 | 400 |
| 40904 | 角色编码重复 | 409 |

**Examples**
```json
// Valid - 编辑权限
PUT /api/v1/roles/10
{ "role_name": "业务员", "data_scope": "SELF",
  "permissions": [
    { "menu_id": "crm.customer", "actions": ["view", "add", "edit", "export"] },
    { "menu_id": "quote",       "actions": ["view", "add", "edit"] },
    { "menu_id": "order",       "actions": ["view_own", "add"] },
    { "menu_id": "order.customer", "actions": ["view_own"] }   // view_all 不在
  ]
}

// Invalid - 删除内置角色
DELETE /api/v1/roles/1   // role_code = admin
// → 40903 "内置角色不可删除"
```

### AC-1.1.3 — 金额阈值与审批路由（FR-1-1-3）

**GIVEN** 系统已初始化三类角色及阈值：
- `salesperson`（业务员）：`amount_threshold = 50000`
- `dept_manager`（部门经理）：`amount_threshold = 200000`
- `gm`（总经理）：`amount_threshold = NULL`（无限额）

业务员「张三」（`user_id=10086`，仅 `salesperson` 一个角色）已登录
**WHEN** 张三提交一笔报价，金额 `60000.00` 元（> 5万，≤ 20万）
**THEN**
1. 系统创建报价单 `BJ202606090001`（单号生成由 Story 1.3 负责，本 Story 仅消费 `POST /api/v1/system/serial/BJ`），`status = PENDING`；
2. **审批路由计算**（本 Story 语义，不实现工作流引擎）：遍历"业务员"角色的 amount_threshold 链，发现 60000 > 50000，**跳过业务员自审**，路由到 `dept_manager` 角色；
3. 取 `dept_manager` 角色下第一个 `status=ACTIVE` 的用户（按 `user_id ASC`）作为审批人 `user_id=10010`；张三"我的待办"看不到该单（`GET /api/v1/approvals/pending` 不返回），`user_id=10010` 的待办返回该单；
4. 阈值常量在 PRD 附录 A（`docs/prd/附录-a金额阈值速查表.md`）"V1.3.7 客户口径"中明确：`< 5万` 业务员自审，`[5万, 20万]` 部门经理，`> 20万` 总经理；
5. 若同时有 2 个部门经理（`dept_manager` 角色下 user_a 与 user_b），取 `user_id` 较小者；支持后续在 Story 1.2 升级为"按部门经理角色全员 OR 自选"。

**Business Rules**
- BR-13 金额阈值定义在角色上（`sys_role.amount_threshold`），可在角色管理中调整
- BR-14 审批路由：金额 < 50000 → salesperson 自审；50000 ≤ 金额 ≤ 200000 → dept_manager；金额 > 200000 → gm
- BR-15 同一审批节点多人时，按"该角色下 `user_id ASC` 第一个有效用户"取（V1.3.7 简化版；Story 1.2 工作流引擎可升级为 OR 会签）
- BR-16 金额比较用 `BigDecimal`，`compareTo` 而非 `equals`（`new BigDecimal("60000.00").compareTo(new BigDecimal("50000")) > 0`）
- BR-17 金额从 `sys_role.amount_threshold` 实时读取，不缓存（角色调整后立即生效）；如需缓存，必须配合 `@CacheEvict` 在 `PUT /api/v1/roles/{id}` 时清空
- BR-18 V1.3.7 移除"代选厂商"红线（不在本 Story 范围）；本 Story 严禁将阈值硬编码到业务代码

**Data Validation**
- `amount_threshold`：可选，`DECIMAL(15,2)`，`> 0`；NULL 表示无限额（如 `gm`）
- 阈值调整时 `old_value` / `new_value` 必须写入 `platform_audit_log`（敏感字段）

**Error Handling**
| 错误码 | 触发条件 | HTTP |
|--------|----------|------|
| 40005 | 角色 threshold 格式错误（非数字/负数） | 400 |
| 40402 | 路由时找不到 `dept_manager` 角色下的有效用户 | 404（必须人工干预） |

**Examples**
```
case 1: amount = 30000, role = salesperson
  → 自审（30000 <= 50000），张三自己出现在"我的待办"

case 2: amount = 60000, role = salesperson
  → 跳过张三，路由到 dept_manager 第一人 user_id=10010
  → 10010 待办，张三不出现

case 3: amount = 250000, role = salesperson
  → 跳过 salesperson + dept_manager，路由到 gm 第一人
  → dept_manager 阈值 200000 不通过 → gm 阈值 NULL 通过
```

### V1.3.7 关键约束（不可妥协）

1. **DEK 密钥管理**（V1.3.6 升级）：`/etc/erp/dek.key` 由**甲方 IT 独立保管**，`chmod 600`，乙方不持有。启动时 `Files.readAllBytes()` 读取，失败 → 拒绝启动。
2. **BCrypt cost = 12**（V1.3.7 沿用）：不要尝试 cost=10/14，统一 12。`BCryptPasswordEncoder(12)` 构造。
3. **JWT TTL**：access = 2h（7200s），refresh = 7d（604800s）；黑名单 TTL = access 剩余有效期，**严禁** 用固定 2h（用户改密后旧 token 立即失效）。
4. **金额阈值**（V1.3.7 客户口径）：`< 5万` / `5-20万` / `> 20万`，写入 `sys_role.amount_threshold`；可调；实时生效（无缓存）。
5. **审计日志保留 ≥ 1 年**（V1.3.7 红线）：`platform_audit_log` 定时清理任务在 Story 1.3 系统参数中配置；本 Story 仅写日志，不实现清理。
6. **5 条 V1.3.7 红线**（**本 Story 不涉及**）：①"代选厂商"工序分配职责分离、②"工序报工去单价"留痕、③质量异议次数预警、④V1.3.4 rework_count 升级、⑤V1.3.7 P0 工单 N 天未启动预警。本 Story 范围仅限用户/角色/权限，不展开。
7. **多租户**：本 Story 暂不实现（V1.3.7 已知缺口，写入 `docs/architecture/11-风险与缓解v137-新增-3-项共-13-项.md` 风险清单）；`sys_user.tenant_id` 字段不预留，后续 Story 1.x 补齐。
